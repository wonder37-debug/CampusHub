# **CampusHub 项目 AI 工具选型评估报告**

**评估日期：** 2026年4月11日

**评估人：** 全体团队成员

**选型目标：** 构建覆盖需求、架构、编码、审查及测试的全链路 AI 协同工具链。

## **1\. AI 编程助手 (AI Coding Assistant)**

| 工具名称 | 评估结论 | 选择/不选择理由 |
| :---- | :---- | :---- |
| **Cursor** | **选定** | **优势：** 支持全仓库索引（Codebase Indexing），能跨文件理解 Monorepo 架构；Composer 模式支持多文件协同编辑。 **易用性：** 极高，适合复杂重构。 |
| **GitHub Copilot** | 备选 | **理由：** 虽稳定性好，但在处理跨文件逻辑引用和 Monorepo 项目结构时的深度理解能力略逊于 Cursor 的索引模式。 |
| **通义灵码 (Lingma)** | 备选/国内替代 | **优势：** 深度适配国内网络，支持研发智能体，对 Java (Spring Boot) 的单测生成采纳率高。 |

## **2\. AI 对话与文档工具 (AI Chat & Docs)**

| 工具名称 | 评估结论 | 选择/不选择理由 |
| :---- | :---- | :---- |
| **Claude 3.5 Sonnet** | **选定** | **优势：** HumanEval 评分极高，逻辑严谨且幻觉率低；具备 200k 超长上下文，极其适合分析复杂的 SRS 需求文档和长篇代码。 |
| **ChatGPT (GPT-4o)** | 备选 | **理由：** 虽然多模态能力强，但在处理复杂代码逻辑和遵循长篇 system prompt 时的连贯性略逊于 Claude 3.5。 |
| **文心一言 / Kimi** | 不选择 | **理由：** 针对技术架构深度分析和特定编码任务的精确度与全球顶尖模型仍有差距。 |

## **3\. AI 代码审查工具 (AI Code Review)**

| 工具名称 | 评估结论 | 选择/不选择理由 |
| :---- | :---- | :---- |
| **CodeRabbit** | **选定** | **优势：** 自动捕捉运行时 Bug 的概率达 46%；提供 PR 自动总结和一行式修复建议。 **成本：** 个人开源版免费，集成于 GitHub Actions。 |
| **AI Code Reviewer (GitHub App)** | 不选择 | **理由：** 建议较为泛化，无法像 CodeRabbit 那样提供上下文相关的深度性能优化建议。 |

## **4\. AI 测试与质量保障 (AI Testing)**

| 工具名称 | 评估结论 | 选择/不选择理由 |
| :---- | :---- | :---- |
| **Qodo (原 CodiumAI)** | **选定** | **优势：** 专注于企业级测试生成，能理解跨仓库依赖；支持自动生成覆盖边界条件的 JUnit 测试模板。 |
| **Diffblue Cover** | 不选择 | **理由：** 虽自动化程度高，但配置较为复杂，对 Vue 3 前端代码的支持不如 Qodo 全面。 |

## **最终选定工具链清单 (Stack Summary)**

| 分类 | 选定工具 | 核心应用场景 |
| :---- | :---- | :---- |
| **编程助手** | **Cursor** | 日常编码、多文件重构、全仓库检索 |
| **对话决策** | **Claude 3.5** | 需求分析、架构设计建议、复杂 Debug |
| **代码审查** | **CodeRabbit** | PR 自动评审、代码规范性检查 |
| **测试增强** | **Qodo** | 自动化单元测试生成、边界值测试 |

## **思考练习反思 (Exercise Insight)**

在尝试利用 AI 生成 README.md 初稿后，我们发现：

1. **通用套话**：AI 生成的技术栈描述和标准 Git 指令非常专业且通用。  
2. **团队特性**：项目特有的“校园互助”业务逻辑（如具体的积分激励机制、校园认证流程）以及本校特有的基础设施配置，必须由团队讨论确定。  
3. **启示**：AI 负责构建“骨架”，人类负责填充“灵魂”。在后续 P1 需求分析阶段，我们将利用 Claude 辅助生成文档，但业务逻辑的准确性必须由郑嘉鸿（需求负责人）进行终审。
