## 3. 按 P3 / P1 吸收的关键约束

### 来自 P3 API 规范

- 接单入口语义与 `accept` 动作保持一致
- 订单详情只允许参与方或管理员查看
- 重复接单返回冲突
- 对并发修改操作预留一致性控制能力

### 来自 P1 需求

- 接单者完成任务时需要提供存证，因此订单模型预留了：
  - `proofSubmitted`
  - `proofImageCount`
- 完成订单时先要求 `proofImageCount` 在 `1-3` 范围内
- 24 小时自动结算、申诉仲裁、积分划拨本轮只做后续占位，不强行并入当前实现

## 4. 本轮状态机设计

当前采用的最小可落地状态机为：

- `ACCEPTED -> IN_PROGRESS`
- `IN_PROGRESS -> COMPLETED`
- `ACCEPTED -> CANCELLED`

补充规则：

- 只有接单者可以把订单推进到 `IN_PROGRESS`
- 只有接单者可以把订单推进到 `COMPLETED`
- 订单完成时要求提供 `1-3` 张成果证明图的数量
- 取消目前只允许发生在 `ACCEPTED` 阶段

## 本轮已落地业务规则

本轮已经写入代码并通过测试的规则包括：

- 发布者不能接自己的需求
- 需求必须处于 `PENDING` 才能接单
- 同一需求不能重复接单
- 接单成功后需求状态变为 `IN_PROGRESS`
- 非参与方不能查看订单详情
- 订单状态不能非法跳转
- 完成订单时必须携带合法的存证图片数量
- 订单历史只返回当前用户参与的订单
