# P4 前后端对接修订规范

## 1. 文档目的

本文档用于对照 [API规范.md](../P3/API规范.md) 与当前后端服务层已落地实现，给前端同学提供一份“当前联调应以什么为准”的修订版规范。

这份文档的目标不是推翻 P3，而是解决两个现实问题：

1. P3 API 规范中有些内容是设计期方案，和当前代码实现存在细微偏差
2. 前端联调阶段更需要一份“现在就能对”的接口与字段口径

结论原则：

- 若 P3 与当前后端实现一致：继续按 P3 执行
- 若 P3 与当前后端实现不一致：当前联调阶段以前端按“本修订规范”对接为准
- 若 P3 中有接口，但当前后端服务层尚未落地：明确标注“暂未实现”

---

## 2. 本次核对结论

整体上，主流程方向与 P3 是一致的，但以下几类差异需要前端特别注意：

- 枚举值传输口径：当前后端更偏向传英文枚举码，不是中文显示名
- 需求模块字段：当前实现比 P3 多了 `campusZone`、`anonymous`
- 列表/推荐筛选：当前实现支持 `campusZone`，并扩展了 `sort=recommend`
- 管理后台统计返回：当前已实现的数据结构比 P3 的概述更具体
- 少数 P3 已写的接口，当前服务层还没有对应应用服务落地

---

## 3. 关键差异总表

| 模块 | P3 写法 | 当前后端实现 | 联调采用口径 |
|---|---|---|---|
| 注册前邮箱验证码 | 已定义 `POST /api/v1/auth/email-code` | 服务层暂未实现 | 标记为暂未实现 |
| 需求分类 `category` | 文档中偏中文展示值 | 后端要求英文枚举码 | 以前端传英文枚举码为准 |
| 发布需求字段 | P3 未明确 `campusZone`、`anonymous` | 当前已要求/支持 | 联调必须带 `campusZone`，可带 `anonymous` |
| 需求排序 `sort` | `time/distance/reward` | 已扩展 `recommend` | 前端可增加 `recommend` |
| 需求状态展示 | P3 更偏业务描述 | 当前返回英文状态码 | 联调按英文状态码，前端自行映射中文 |
| 管理后台待审核状态 | P3 定义“待审核需求” | 当前内部状态为 `REVIEWING` | 以前端识别 `REVIEWING` 为待审核 |
| 审核驳回结果 | P3 未细化最终状态 | 当前落到 `CANCELLED` | 联调按 `CANCELLED` 理解为已驳回/不可继续流转 |
| 推荐接口 | P3 有 `GET /api/v1/recommendations` | 服务层已实现 | 可联调 |
| 通知已读 | P3 只写了列表 | 服务层已有 `markAsRead` 能力 | 可提前预留接口 |

## 4. 修订后的前端对接规范

## 4.1 通用约定

### 基础路径

- 统一前缀：`/api/v1`

### 鉴权

- 受保护接口统一使用：

```http
Authorization: Bearer <token>
```

### 通用成功结构

```json
{
  "code": 0,
  "message": "OK",
  "data": {}
}
```

### 通用失败结构

```json
{
  "code": 1002,
  "message": "参数校验失败",
  "errors": {}
}
```

### 分页结构

统一按以下结构理解：

```json
{
  "items": [],
  "page": 1,
  "size": 20,
  "total": 123
}
```

其中：

- `total` 表示命中条件后的总条数
- 不是当前页条数

---

## 4.2 枚举与状态码口径

### 需求分类 `category`

前端请求时请传英文枚举码，不要直接传中文：

- `EXPRESS`
- `STUDY_TUTORING`
- `SECOND_HAND`
- `TEAM_UP`
- `OTHER`

前端展示层如需中文，请自行映射：

- `EXPRESS` -> 取快递
- `STUDY_TUTORING` -> 学习辅导
- `SECOND_HAND` -> 二手交易
- `TEAM_UP` -> 活动组队
- `OTHER` -> 其他

### 校区 `campusZone`

当前需求发布/查询已使用英文枚举码，前端应传枚举值，不传中文名。

当前代码中至少已使用：

- `XIANLIN`

说明：

- 若前端需要完整校区选项，请以后端枚举源码或后续补充字典接口为准

### 需求状态 `status`

后端当前返回英文状态码：

- `PENDING`
- `REVIEWING`
- `IN_PROGRESS`
- `COMPLETED`
- `CANCELLED`

前端建议映射：

- `PENDING` -> 待接单 / 已发布
- `REVIEWING` -> 审核中
- `IN_PROGRESS` -> 进行中
- `COMPLETED` -> 已完成
- `CANCELLED` -> 已取消 / 已驳回

### 订单状态 `status`

后端当前返回英文状态码：

- `ACCEPTED`
- `IN_PROGRESS`
- `COMPLETED`
- `CANCELLED`

### 用户角色 `role`

- `USER`
- `ADMIN`

### 用户状态 `status`

- `ACTIVE`
- `BANNED`

### 通知类型 `type`

- `ORDER_ACCEPTED`
- `STATUS_CHANGED`
- `REVIEW_RECEIVED`

## 4.3 认证模块

### 1. 邮箱验证码接口

P3 定义：

- `POST /api/v1/auth/email-code`

当前状态：

- 暂未在服务层落地

联调结论：

- 前端可以先保留页面交互
- 但当前不要依赖该接口实际可用

### 2. 注册

- `POST /api/v1/auth/register`

请求体当前可对齐：

```json
{
  "email": "user@example.edu.cn",
  "verificationCode": "123456",
  "studentId": "20260001",
  "password": "Password1",
  "nickname": "张三",
  "avatarUrl": "https://..."
}
```

说明：

- 当前后端已经要求 `verificationCode`
- 但验证码发送接口还未落地，因此测试阶段通常需要后端同学配合或临时跳过真实发送

### 3. 登录

- `POST /api/v1/auth/login`

请求体：

```json
{
  "loginId": "20260001",
  "password": "Password1"
}
```

返回体：

```json
{
  "token": "<jwt>",
  "expiresIn": 3600,
  "user": {
    "id": 1,
    "email": "user@example.edu.cn",
    "studentId": "20260001",
    "nickname": "张三",
    "avatarUrl": null,
    "role": "USER",
    "status": "ACTIVE",
    "creditScore": 100
  }
}
```

## 4.4 需求模块

### 1. 发布需求

- `POST /api/v1/demands`

修订后的请求体：

```json
{
  "title": "取快递",
  "description": "帮忙去驿站取一下",
  "category": "EXPRESS",
  "campusZone": "XIANLIN",
  "location": "仙林校区菜鸟驿站",
  "startTime": "2026-05-11T18:00:00",
  "endTime": "2026-05-11T20:00:00",
  "reward": 2.5,
  "tags": ["快递", "代取"],
  "anonymous": false
}
```

与 P3 差异：

- 当前联调必须把 `category` 当成英文枚举码
- 当前联调必须带 `campusZone`
- 当前实现支持 `anonymous`

### 2. 需求列表

- `GET /api/v1/demands`

当前推荐查询参数：

- `q`
- `category`
- `campusZone`
- `location`
- `startTimeFrom`
- `startTimeTo`
- `sort`
- `page`
- `size`

其中 `sort` 当前支持：

- `TIME`
- `DISTANCE`
- `REWARD`
- `RECOMMEND`

前端联调建议：

- 如果是 URL 查询参数层，优先传小写或后端网关统一转换前，建议直接按后端预期约定传枚举含义值
- 为避免歧义，当前联调文档中建议前端统一传：
  - `time`
  - `distance`
  - `reward`
  - `recommend`

说明：

- 服务层内部会转为枚举
- `recommend` 是当前实现新增能力，P3 原普通列表段落未写，但推荐功能与当前后端已支持

### 3. 需求详情

- `GET /api/v1/demands/{demandId}`

当前返回中需要前端注意：

- `status` 是英文状态码
- 匿名需求下：
  - `publisherId` 可能为 `null`
  - `publisherDisplayName` 会显示匿名码
  - `anonymousCode` 单独返回

## 4.5 订单模块

### 1. 接单

- `POST /api/v1/demands/{demandId}/accept`

请求体：

```json
{
  "note": "我可以现在去取"
}
```

返回重点：

- 返回的是订单详情结构
- `status` 初始为 `ACCEPTED`
- 需求状态会同步变成 `IN_PROGRESS`

### 2. 订单详情

- `GET /api/v1/orders/{orderId}`

当前后端返回字段中，前端需要重点使用：

- `orderId`
- `status`
- `demandId`
- `publisherId`
- `accepterId`
- `acceptNote`
- `proofSubmitted`
- `proofImageCount`
- `createdAt`
- `updatedAt`
- `completedAt`
- `demand`
- `statusHistory`

### 3. 订单状态更新

P3 当前详细 API 中没有完全展开，但当前后端服务层已具备能力，前端应预留：

- `targetStatus`
- `note`
- `proofImageCount`

当前允许的主要流转：

- `ACCEPTED -> IN_PROGRESS`
- `IN_PROGRESS -> COMPLETED`
- `ACCEPTED -> CANCELLED`

## 4.6 评价模块

### 1. 提交评价

- `POST /api/v1/orders/{orderId}/reviews`

请求体：

```json
{
  "rating": 5,
  "comment": "非常靠谱"
}
```

当前已实现约束：

- 只能订单参与方评价
- 只能已完成订单评价
- 同一订单同一评价人只能评价一次

### 2. 评价列表

P3 主文档未重点展开，但当前后端服务层已有 `listUserReviews(...)` 能力，前端可以提前预留“用户评价列表”页面能力。

## 4.7 通知模块

### 1. 通知列表

- `GET /api/v1/notifications`

查询参数：

- `unreadOnly`
- `page`
- `size`

返回项当前包括：

- `id`
- `type`
- `title`
- `content`
- `read`
- `relatedId`
- `createdAt`

### 2. 标记已读

P3 当前主文档未列出，但当前后端服务层已有能力：

- `markAsRead(Long userId, Long notificationId)`

联调建议：

- 前端可以提前预留 “标记已读” 按钮和交互
- 后续 Controller 落地时建议接口形式为：
  - `POST /api/v1/notifications/{notificationId}/read`
  - 或 `PATCH /api/v1/notifications/{notificationId}`

当前以“能力已存在、HTTP 形式待 Controller 确认”标注

## 4.8 管理后台

### 1. 用户列表

- `GET /api/v1/admin/users`

当前建议查询参数：

- `q`
- `page`
- `size`

### 2. 封禁 / 解封

- `POST /api/v1/admin/users/{userId}/ban`
- `POST /api/v1/admin/users/{userId}/unban`

当前封禁规则：

- 管理员不能封禁自己

### 3. 待审核需求

- `GET /api/v1/admin/demands/pending`

当前建议查询参数：

- `q`
- `category`
- `page`
- `size`

当前实现中，“待审核”对应状态：

- `REVIEWING`

### 4. 审核需求

- `POST /api/v1/admin/demands/{demandId}/review`

请求体：

```json
{
  "action": "approve",
  "reason": "内容合规"
}
```

当前实现口径：

- `approve` -> 需求状态变 `PENDING`
- `reject` -> 需求状态变 `CANCELLED`

前端展示建议：

- `CANCELLED` 在后台审核语境下可展示为“已驳回”

### 5. 统计看板

- `GET /api/v1/admin/stats`

当前后端已实现的 `data` 结构：

```json
{
  "dailyActiveUsers": 10,
  "totalUsers": 100,
  "totalDemands": 80,
  "pendingReviewDemands": 5,
  "totalOrders": 40,
  "completedOrders": 30,
  "categoryDistribution": [
    {
      "category": "EXPRESS",
      "count": 12
    }
  ]
}
```

与 P3 差异：

- P3 只概述了“日活、订单数、分类别统计”
- 当前后端返回结构已经具体化，前端应按当前字段名对接

## 4.9 推荐模块

### 1. 推荐列表

- `GET /api/v1/recommendations`

当前已实现约束：

- 需登录
- `size <= 50`

返回结构当前可按：

```json
{
  "items": [
    {
      "demandId": 1,
      "score": 0.9,
      "rank": 1,
      "reasonTags": ["同分类", "历史接单偏好"]
    }
  ],
  "page": 1,
  "size": 10,
  "total": 20
}
```

说明：

- 当前推荐结果项不直接携带完整需求摘要
- 如果前端推荐页需要展示更丰富的需求信息，有两种做法：
  - 用 `demandId` 再查详情
  - 后续由 Controller 层改成组合返回

这是当前前后端联调中需要提前确认的点。

## 5. 当前明确存在的前后端差异

以下是最值得前端同学提前知道的“不要踩坑”点：

1. `category` 不要传中文，传英文枚举码。
2. 发布需求时不要漏 `campusZone`。
3. 状态字段统一按英文码处理，页面再做中文映射。
4. `REVIEWING` 是待审核，不是普通用户可见的开放状态。
5. 后台审核驳回目前落成 `CANCELLED`，页面要按业务语义解释，不要简单显示“用户取消”。
6. 推荐接口当前只返回 `demandId + score + rank + reasonTags`，不是完整需求卡片。
7. 邮箱验证码发送接口在 P3 里有，但当前服务层尚未真正落地。

---

## 6. 建议前端同学的联调顺序

建议按以下顺序联调：

1. 登录
2. 需求发布 / 列表 / 详情
3. 接单 / 订单详情
4. 评价 / 通知
5. 推荐
6. 管理后台

原因：

- 主链路先通，能尽快发现枚举与状态流转问题
- 后台和推荐更依赖前面数据闭环

---

## 7. 对接完成标准

前后端对接时，建议至少满足：

1. 所有枚举字段都按英文码传输
2. 分页结构统一理解为 `items/page/size/total`
3. 状态流转展示不再把后台状态和用户侧状态混淆
4. 推荐页能够正确处理“只有 `demandId` 没有完整详情”的当前实现
5. 管理后台页面能正确区分 `REVIEWING`、`PENDING`、`CANCELLED`

---

## 8. 备注

这份修订规范的本质，是把“设计期接口描述”更新为“当前代码可联调口径”。后续如果 Controller/OpenAPI 层落地后又统一了一轮字段名或返回结构，应再以正式 OpenAPI 文档为最终准绳；但在当前阶段，这份文档更适合作为前端同学直接使用的联调依据。
