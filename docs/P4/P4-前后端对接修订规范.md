# P4 前后端对接修订规范

## 1. 文档目的

本文档用于对照 [API规范.md](../P3/API规范.md) 与当前后端服务层已落地实现，给前端同学提供一份“当前联调应以什么为准”的修订版规范。

这份文档的目标不是推翻 P3，而是解决两个现实问题：

1. P3 API 规范中有些内容是设计期方案，和当前代码实现存在细微偏差
2. 前端联调阶段更需要一份“现在就能对”的接口与字段口径

结论原则：

- 若 P3 与当前后端实现一致：继续按 P3 执行
- 若 P3 与当前后端实现不一致：当前联调阶段以前端按“本修订规范”对接为准
- 若 P3 中有接口，但当前后端服务层尚未落地：明确标注“暂未实现”

---

## 2. 本次核对结论

整体上，主流程方向与 P3 是一致的，但以下几类差异需要前端特别注意：

- 枚举值传输口径：当前后端更偏向传英文枚举码，不是中文显示名
- 需求模块字段：当前实现比 P3 多了 `campusZone`、`anonymous`
- 列表/推荐筛选：当前实现支持 `campusZone`，并扩展了 `sort=recommend`
- 管理后台统计返回：当前已实现的数据结构比 P3 的概述更具体
- 少数 P3 已写的接口，当前服务层还没有对应应用服务落地

---

## 3. 关键差异总表

| 模块 | P3 写法 | 当前后端实现 | 联调采用口径 |
|---|---|---|---|
| 注册前邮箱验证码 | 已定义 `POST /api/v1/auth/email-code` | 服务层暂未实现 | 标记为暂未实现 |
| 需求分类 `category` | 文档中偏中文展示值 | 后端要求英文枚举码 | 以前端传英文枚举码为准 |
| 发布需求字段 | P3 未明确 `campusZone`、`anonymous` | 当前已要求/支持 | 联调必须带 `campusZone`，可带 `anonymous` |
| 需求排序 `sort` | `time/distance/reward` | 已扩展 `recommend` | 前端可增加 `recommend` |
| 需求状态展示 | P3 更偏业务描述 | 当前返回英文状态码 | 联调按英文状态码，前端自行映射中文 |
| 管理后台待审核状态 | P3 定义“待审核需求” | 当前内部状态为 `REVIEWING` | 以前端识别 `REVIEWING` 为待审核 |
| 审核驳回结果 | P3 未细化最终状态 | 当前落到 `CANCELLED` | 联调按 `CANCELLED` 理解为已驳回/不可继续流转 |
| 推荐接口 | P3 有 `GET /api/v1/recommendations` | 服务层已实现 | 可联调 |
| 通知已读 | P3 只写了列表 | 服务层已有 `markAsRead` 能力 | 可提前预留接口 |

## 4. 修订后的前端对接规范

## 4.1 通用约定

### 基础路径

- 统一前缀：`/api/v1`

### 鉴权

- 受保护接口统一使用：

```http
Authorization: Bearer <token>
```

### 通用成功结构

```json
{
  "code": 0,
  "message": "OK",
  "data": {}
}
```

### 通用失败结构

```json
{
  "code": 1002,
  "message": "参数校验失败",
  "errors": {}
}
```

### 分页结构

统一按以下结构理解：

```json
{
  "items": [],
  "page": 1,
  "size": 20,
  "total": 123
}
```

其中：

- `total` 表示命中条件后的总条数
- 不是当前页条数

---

## 4.2 枚举与状态码口径

### 需求分类 `category`

前端请求时请传英文枚举码，不要直接传中文：

- `EXPRESS`
- `STUDY_TUTORING`
- `SECOND_HAND`
- `TEAM_UP`
- `OTHER`

前端展示层如需中文，请自行映射：

- `EXPRESS` -> 取快递
- `STUDY_TUTORING` -> 学习辅导
- `SECOND_HAND` -> 二手交易
- `TEAM_UP` -> 活动组队
- `OTHER` -> 其他

### 校区 `campusZone`

当前需求发布/查询已使用英文枚举码，前端应传枚举值，不传中文名。

当前代码中至少已使用：

- `XIANLIN`

说明：

- 若前端需要完整校区选项，请以后端枚举源码或后续补充字典接口为准

### 需求状态 `status`

后端当前返回英文状态码：

- `PENDING`
- `REVIEWING`
- `IN_PROGRESS`
- `COMPLETED`
- `CANCELLED`

前端建议映射：

- `PENDING` -> 待接单 / 已发布
- `REVIEWING` -> 审核中
- `IN_PROGRESS` -> 进行中
- `COMPLETED` -> 已完成
- `CANCELLED` -> 已取消 / 已驳回

### 订单状态 `status`

后端当前返回英文状态码：

- `ACCEPTED`
- `IN_PROGRESS`
- `COMPLETED`
- `CANCELLED`

### 用户角色 `role`

- `USER`
- `ADMIN`

### 用户状态 `status`

- `ACTIVE`
- `BANNED`

### 通知类型 `type`

- `ORDER_ACCEPTED`
- `STATUS_CHANGED`
- `REVIEW_RECEIVED`

## 4.4 需求模块

### 1. 发布需求

- `POST /api/v1/demands`

修订后的请求体：

```json
{
  "title": "取快递",
  "description": "帮忙去驿站取一下",
  "category": "EXPRESS",
  "campusZone": "XIANLIN",
  "location": "仙林校区菜鸟驿站",
  "startTime": "2026-05-11T18:00:00",
  "endTime": "2026-05-11T20:00:00",
  "reward": 2.5,
  "tags": ["快递", "代取"],
  "anonymous": false
}
```

与 P3 差异：

- 当前联调必须把 `category` 当成英文枚举码
- 当前联调必须带 `campusZone`
- 当前实现支持 `anonymous`

### 2. 需求列表

- `GET /api/v1/demands`

当前推荐查询参数：

- `q`
- `category`
- `campusZone`
- `location`
- `startTimeFrom`
- `startTimeTo`
- `sort`
- `page`
- `size`

其中 `sort` 当前支持：

- `TIME`
- `DISTANCE`
- `REWARD`
- `RECOMMEND`

前端联调建议：

- 如果是 URL 查询参数层，优先传小写或后端网关统一转换前，建议直接按后端预期约定传枚举含义值
- 为避免歧义，当前联调文档中建议前端统一传：
  - `time`
  - `distance`
  - `reward`
  - `recommend`

说明：

- 服务层内部会转为枚举
- `recommend` 是当前实现新增能力，P3 原普通列表段落未写，但推荐功能与当前后端已支持

### 3. 需求详情

- `GET /api/v1/demands/{demandId}`

当前返回中需要前端注意：

- `status` 是英文状态码
- 匿名需求下：
  - `publisherId` 可能为 `null`
  - `publisherDisplayName` 会显示匿名码
  - `anonymousCode` 单独返回

## 5. 当前明确存在的前后端差异

以下是最值得前端同学提前知道的“不要踩坑”点：

1. `category` 不要传中文，传英文枚举码。
2. 发布需求时不要漏 `campusZone`。
3. 状态字段统一按英文码处理，页面再做中文映射。
4. `REVIEWING` 是待审核，不是普通用户可见的开放状态。
5. 后台审核驳回目前落成 `CANCELLED`，页面要按业务语义解释，不要简单显示“用户取消”。
6. 推荐接口当前只返回 `demandId + score + rank + reasonTags`，不是完整需求卡片。
7. 邮箱验证码发送接口在 P3 里有，但当前服务层尚未真正落地。

---

## 6. 建议前端同学的联调顺序

建议按以下顺序联调：

1. 登录
2. 需求发布 / 列表 / 详情
3. 接单 / 订单详情
4. 评价 / 通知
5. 推荐
6. 管理后台

原因：

- 主链路先通，能尽快发现枚举与状态流转问题
- 后台和推荐更依赖前面数据闭环

---

## 7. 对接完成标准

前后端对接时，建议至少满足：

1. 所有枚举字段都按英文码传输
2. 分页结构统一理解为 `items/page/size/total`
3. 状态流转展示不再把后台状态和用户侧状态混淆
4. 推荐页能够正确处理“只有 `demandId` 没有完整详情”的当前实现
5. 管理后台页面能正确区分 `REVIEWING`、`PENDING`、`CANCELLED`

## 9. 2026-05-26 后端服务层新增联调修订

本节用于同步 2026-05-26 这一轮后端服务层修复后，前端需要立即对齐的行为变化。以下内容优先级高于前文中较早阶段的暂定口径。

### 9.1 需求大厅与需求详情

- `GET /api/v1/demands`

本轮修订后，公开需求列表的服务层行为已明确为：

- 仅返回 `status = PENDING` 的需求
- 自动排除已过期需求
- `REVIEWING`、`IN_PROGRESS`、`COMPLETED`、`CANCELLED` 不应再出现在首页大厅

前端联调要求：

- 不要再假设审核中的需求会出现在大厅列表
- 大厅筛选结果为空时，应按“当前无可公开需求”处理，而不是认为接口异常
- 对于“已过期”需求，当前服务层已做过滤，但尚未单独暴露 `EXPIRED` 状态码给前端

### 9.2 发布需求的金额校验

- `POST /api/v1/demands`
- `PUT /api/v1/demands/{demandId}`

本轮修订后，后端服务层新增余额约束：

- `reward` 不能超过当前用户可用余额
- 可用余额按 `balance - frozenBalance` 理解

联调影响：

- 前端提交有偿需求时，应预期接口可能返回余额不足类校验错误
- 当用户余额不足时，发布和编辑调高报酬都可能失败
- 免费需求 `reward = 0` 不受影响

建议前端提示文案：

- “报酬不能超过当前可用余额”

### 9.3 管理员账号的交易限制

本轮修订后，后端服务层已收紧管理员权限边界：

- 管理员不能发布需求
- 管理员不能接单

联调影响：

- 前端若检测到当前用户 `role = ADMIN`，应隐藏或禁用“发布需求”“立即接单”等普通交易入口
- 即使前端未隐藏，后端也会拒绝请求，因此不要将其当成普通失败重试

### 9.4 需求发布后的管理员通知

本轮修订后，用户发布需求成功后，后端会向所有管理员创建站内通知。

当前联调口径：

- 通知 `type` 仍然复用 `STATUS_CHANGED`
- `title` 为 `需求待审核`
- `relatedId` 指向新发布的需求 ID

前端处理建议：

- 在管理员通知页中，如果遇到 `type = STATUS_CHANGED` 且标题为 `需求待审核`，应按“新需求待审核通知”处理
- 点击后建议跳转到后台审核列表，并尽量定位到对应 `relatedId`

说明：

- 这是一版兼容现有后端结构的过渡方案
- 当前尚未新增独立的 `REVIEW_REQUEST` 枚举值，因此前端请勿等待新类型再接入

### 9.5 订单完成流程口径确认

本轮服务层与前端联调时，需要继续按“双确认完成”理解订单流转：

- 接单方提交 `COMPLETED` 时：
  - 订单最终状态仍可能是 `IN_PROGRESS`
  - 但会记录 `proofSubmitted = true`
  - 并保留 `proofImageCount`
- 发布方再次确认 `COMPLETED` 后：
  - 订单状态才真正变为 `COMPLETED`

前端联调要求：

- 不要把“接单方第一次提交完成”误显示成订单已结束
- 应根据 `status + proofSubmitted + statusHistory` 区分“待对方确认完成”和“真正已完成”

### 9.6 信用分展示口径更新

本轮修订后，信用分已不再使用“评分平均值乘以 20”的陡降算法，而改为平滑更新。

联调影响：

- 前端不要再假设：
  - `5` 星一定立即映射为 `100`
  - `4` 星一定立即映射为 `80`
  - `1` 星一定立即映射为 `20`
- 当前应始终以后端收到的 `creditScore` 实际值为准显示

示例：

- 默认 `100` 分用户收到一次 `4` 星评价后，当前服务层结果为 `98`

### 9.7 个人资料昵称唯一性

- 资料更新接口

本轮修订后，后端服务层新增昵称唯一性校验：

## 11. 2026-05-26 晚间补充：通知联调口径

本节用于同步“通知模板统一收口”后的联调要求。若与前文旧的过渡性说明冲突，以本节为准。

### 11.1 通知类型口径

当前前端需要按以下 `type` 处理通知：

- `ORDER_ACCEPTED`
- `STATUS_CHANGED`
- `REVIEW_RECEIVED`
- `REVIEW_REQUEST`
- `DEMAND_REJECTED`

其中：

- 管理员待审核通知，优先按 `REVIEW_REQUEST` 识别。
- 需求驳回通知，优先按 `DEMAND_REJECTED` 识别。
- 不再建议依赖“`STATUS_CHANGED + title=需求待审核`”这种旧过渡规则。

### 11.2 通知正文口径

当前订单接单、订单状态更新、待确认完成、收到评价、需求待审核、需求驳回等通知，标题和正文都已由后端统一模板生成。

前端联调时请注意：

- 不要再根据旧的中文正文猜测业务来源。
- 优先使用 `type`、`targetType`、`targetId`、`actionHint` 做跳转和交互判断。
- `title + content` 可以直接作为站内信展示内容，不需要前端二次拼接通知文案。

### 11.3 跳转优先级建议

建议前端按以下优先级处理通知点击：

1. `actionHint`
2. `targetType + targetId`
3. 兜底跳转到对应列表页

建议映射：

- `VIEW_ORDER` -> 订单详情
- `VIEW_ORDER_REVIEWS` -> 订单详情评价区
- `REVIEW_DEMAND` -> 管理后台审核页或待审核需求详情
- `VIEW_DEMAND` -> 需求详情或“我的需求”

### 11.4 本轮前端需要同步的直接影响

- 管理员通知列表中，待审核消息应正式按 `REVIEW_REQUEST` 处理。
- 需求发布者收到驳回通知时，可直接展示后端返回的审核原因，不需要前端再拼接。
- 订单相关通知文案已经稳定，前端不应再为不同调用来源维护多套同类通知标题。
## 2026-06-17 增量修订

以下内容为本次新增接口，优先级高于文中旧描述，前后端联调请直接按这里执行。

### 认证模块

- `POST /api/v1/auth/password-reset/code`
  - 作用：向已注册账号邮箱发送找回密码验证码
  - 请求体：

```json
{
  "email": "user@example.edu.cn"
}
```

  - 返回：

```json
{
  "code": 0,
  "message": "OK",
  "data": {
    "expiresIn": 300
  }
}
```

- `POST /api/v1/auth/password-reset`
  - 作用：验证码校验通过后，直接重置密码
  - 请求体：

```json
{
  "email": "user@example.edu.cn",
  "verificationCode": "123456",
  "newPassword": "NewPassword1"
}
```

### 订单模块

- 订单状态新增枚举值：`IN_ARBITRATION`
  - 前端建议展示文案：`仲裁中`

- `POST /api/v1/orders/{orderId}/arbitration`
  - 作用：订单参与方在 `ACCEPTED` 或 `IN_PROGRESS` 状态下发起仲裁
  - 请求体：

```json
{
  "reason": "双方对履约结果存在争议"
}
```

  - 返回：最新 `OrderView`

### 管理员模块

- `DELETE /api/v1/admin/orders/{orderId}`
  - 作用：管理员删除订单
  - 请求体可为空；如需传说明，可使用：

```json
{
  "reason": "违规订单清理"
}
```

- `POST /api/v1/admin/orders/{orderId}/arbitration/resolve`
  - 作用：管理员处理仲裁并决定最终结果
  - `outcome` 目前支持：
    - `complete`：判定订单完成
    - `cancel`：判定订单取消
  - 请求体：

```json
{
  "outcome": "complete",
  "reason": "核实聊天记录后判定接单方已履约"
}
```

  - 返回：最新 `OrderView`

### 通知类型补充

- 新增通知枚举：
  - `ORDER_ARBITRATION_REQUESTED`
  - `ORDER_ARBITRATION_RESOLVED`

### 联调提醒

- 找回密码验证码与注册验证码复用同一套邮箱发送能力，默认有效期返回字段为 `expiresIn`。
- 仲裁发起后，订单普通状态流转应停止，直到管理员裁决。
- 管理员裁决完成后，订单状态会变为 `COMPLETED` 或 `CANCELLED`，前端无需额外推导。
