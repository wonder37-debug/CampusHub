## 本次完成内容

## 1. 新增 recommendation 模块骨架

已新增：

- `recommendation/domain/RecommendationItem`
- `recommendation/dto/RecommendationItemResponse`
- `recommendation/service/RecommendationSwitch`
- `recommendation/service/DefaultRecommendationSwitch`
- `recommendation/service/RecommendationApplicationService`

说明：

- 当前推荐开关通过 `RecommendationSwitch` 抽象，便于后续按配置、A/B 实验或环境变量切换
- 当前版本不落独立推荐表，完全基于现有需求、订单数据在线计算

## 2. 推荐规则第一版已落地

本轮服务层已经支持：

- 仅面向当前用户返回推荐需求
- 只推荐可接单的 `PENDING` 需求
- 自动排除当前用户自己发布的需求
- 自动排除已经被接单、已进入订单流转的需求
- 基于“当前用户历史已接单分类”计算偏好强度
- 对命中偏好分类的需求给出更高分数与解释标签
- 在无历史数据或推荐开关关闭时平滑退化为按时间倒序

当前规则取舍：

- 第一版优先采用“历史接单分类偏好 + 发布时间兜底”的轻量规则
- 暂不引入位置距离、时间窗权重、收藏行为、点击行为

## 3. 推荐结果输出已对齐 P3 API 语义

当前推荐结果已包含：

- `demandId`
- `score`
- `rank`
- `reasonTags`

并已对齐以下约束：

- 推荐分页 `size <= 50`
- 查询条件继续复用 `DemandQuery`
- 推荐只做排序增强，不改变原有筛选条件语义

## 4. 本轮顺手修正的排序问题

在联调测试中额外发现并修正了一个比较器问题：

- 原推荐排序比较器在链式 `reversed()` 使用上会把整个比较链反转，导致低分项有机会排到前面
- 现已改为“分值倒序 + 创建时间倒序”的明确比较逻辑

同时顺手同步修正了需求列表中 `REWARD` 排序的同类写法，避免后续出现“金额排序方向正确但次级时间排序整体反转”的隐患。

## 5. 推荐分页总数字段已修正

本轮还修正了推荐分页返回中的 `total` 语义：

- 修正前：`total` 取的是“分页后的条数”
- 修正后：`total` 取的是“命中筛选与推荐规则后的总条数”

这样可以与项目中现有 `PageResponse` 语义保持一致，也更符合后续 Controller 和前端分页联调预期。

## 按 P3 / P1 吸收的关键约束

### 来自 P3 API 规范

- `GET /api/v1/recommendations`
- 推荐结果包含 `score`、`rank`、`reasonTags`
- 推荐列表分页上限为 `size <= 50`
- 允许通过规则推荐与普通列表排序形成衔接

### 来自 P1 需求

- 推荐的目标是帮助用户更快看到相关任务，尤其是匹配历史偏好的任务类别
- 当前版本先满足“分类相关性优先”的基本体验，不提前引入重型推荐实现

---

## 本轮已落地业务规则

本轮已经写入代码并通过测试的规则包括：

- 推荐前必须校验用户存在
- 推荐查询不能为空
- 推荐分页 `size` 不能超过 `50`
- 仅 `PENDING` 状态需求进入推荐候选
- 已被接单的需求不再进入推荐候选
- 当前用户自己发布的需求不进入推荐候选
- 有历史偏好的用户，优先看到同分类需求
- 无历史偏好或推荐开关关闭时，退化为时间倒序
- 推荐结果中的 `rank` 按最终排序结果顺序生成
