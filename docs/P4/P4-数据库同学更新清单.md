# P4 数据库同学更新清单

## 1. 目的

这份清单用于同步 2026-05-26 这一轮后端服务层收尾后，数据库侧需要确认的内容。

这次重点不是新增很多表，而是确认现有表结构、枚举约束、初始化脚本和测试脚本已经与服务层行为保持一致。

---

## 2. 本轮结论

- 通知模板统一收口属于服务层改动，不新增通知表字段。
- 需求审核闭环、通知类型扩展、需求分类扩展，已经对数据库约束提出明确要求。
- 数据库同学这轮最重要的工作是“核对和同步脚本”，而不是重新设计模型。

---

## 3. 必须核对的表

### 3.1 `sys_notification`

请确认正式库、初始化脚本、迁移脚本中的通知类型约束保持一致。

当前应支持：

- `ORDER_ACCEPTED`
- `STATUS_CHANGED`
- `REVIEW_RECEIVED`
- `REVIEW_REQUEST`
- `DEMAND_REJECTED`

当前服务层仍使用这些字段落库：

- `type`
- `title`
- `content`
- `related_id`
- `is_read`
- `created_at`

说明：

- 这轮统一了通知标题和正文模板，但没有新增列。
- 如果正式库的 `chk_notify_type` 还没有补齐 `REVIEW_REQUEST` 或 `DEMAND_REJECTED`，需要优先处理。

### 3.2 `ord_demand`

请确认需求表已包含以下审核闭环字段：

- `is_approved`
- `reviewed_by`
- `reviewed_at`
- `review_reason`

同时确认约束：

- `status` 允许值覆盖：
  - `PENDING`
  - `REVIEWING`
  - `IN_PROGRESS`
  - `COMPLETED`
  - `CANCELLED`
- `category` 允许值覆盖：
  - `EXPRESS`
  - `ERRAND`
  - `STUDY_TUTORING`
  - `SECOND_HAND`
  - `TEAM_UP`
  - `OTHER`

说明：

- `ERRAND` 已被服务层实际使用。
- `review_reason` 已经不是可有可无的展示字段，而是审核驳回闭环的一部分。

### 3.3 `ord_order`

请确认正式库中以下字段存在：

- `accept_note`
- `proof_submitted`
- `proof_image_count`
- `completed_at`

说明：

- 双确认完成流程依赖这些字段稳定存在。
- 这轮没有新增订单表列，但旧环境如果缺列，会直接影响服务层行为。

### 3.4 `ord_review`

请确认唯一约束仍然存在：

- `uk_review_order_author (order_id, author_id)`

说明：

- 服务层已经做了重复评价拦截。
- 数据库唯一约束仍然必须保留，作为最终防线。

## 4. 需要同步核对的脚本

建议数据库同学至少对照以下文件：

- [init_schema.sql](/E:/homeworki/sec-ii-2026/backend/src/main/resources/init_schema.sql)
- [schema-notification.sql](/E:/homeworki/sec-ii-2026/backend/src/test/resources/schema-notification.sql)
- [schema-demand.sql](/E:/homeworki/sec-ii-2026/backend/src/test/resources/schema-demand.sql)

核对重点：

- 正式库脚本和测试脚本中的枚举取值不要漂移。
- 新增字段、约束如果只在测试脚本存在，正式库也要补齐。
- 如果共享开发库或测试库还没同步，请补迁移 SQL，而不是只改初始化脚本。

## 5. 数据库侧直接行动项

1. 核对 `sys_notification.type` 是否已支持 `REVIEW_REQUEST`、`DEMAND_REJECTED`。
2. 核对 `ord_demand.category` 是否已支持 `ERRAND`。
3. 核对 `ord_demand.reviewed_by / reviewed_at / review_reason / is_approved` 是否都已存在。
4. 核对 `ord_order.proof_submitted / proof_image_count / completed_at` 是否都已存在。
5. 核对 `ord_review` 的 `(order_id, author_id)` 唯一约束是否存在。
6. 若联调库或共享库未同步，请补一份执行顺序明确的迁移清单。

---

## 6. 这轮不需要新增的字段

以下内容本轮数据库侧不需要额外新增列：

- 通知正文模板统一
- 通知响应里的 `targetType`
- 通知响应里的 `targetId`
- 通知响应里的 `targetTitle`
- 通知响应里的 `actionHint`

原因：

- 这些是服务层聚合或响应层映射，不是数据库新增字段。
