## 6. 总体实现原则

## 6.1 模块边界

基于 P2 设计，后端整体虽然是单体工程，但逻辑上要保持以下边界：

- `auth` 负责用户身份、注册登录、资料读取与更新
- `demand` 负责需求发布、查询、详情、可见性与状态基础规则
- `order` 负责接单、状态机、订单历史
- `review` 负责完成后评价与信用分计算
- `notification` 负责站内通知记录生成与读取
- `recommendation` 负责非核心链路的排序增强
- `admin` 负责后台审核、封禁、统计

## 6.2 分层职责

- Controller：参数接收、鉴权入口、返回值包装
- Application Service：主业务流程编排、权限判断、状态流转、事务边界
- Domain：状态规则、枚举、实体行为、纯业务校验
- Repository Interface：面向服务层暴露查询/保存能力
- Repository Impl / Mapper：数据库落地，当前先留空

## 6.3 当前优先策略

编码顺序必须优先保证依赖链正确：

1. 公共返回结构、异常体系、用户上下文、基础枚举
2. `auth` 服务
3. `demand` 服务
4. `order` 服务
5. `review` 与 `notification`
6. `recommendation`
7. `admin`

原因：

- `order` 依赖 `auth` 和 `demand`
- `review` 依赖 `order`
- `notification` 需要被 `order/review/admin` 复用
- `recommendation` 与 `admin` 都依赖前面模块提供的基础查询能力

## 7. 接口契约与公共约束

## 7.1 API 基础约束

根据 P3 API 规范：

- 基础路径：`/api/v1`
- 受保护接口：使用 `Authorization: Bearer <token>`
- 通用成功响应：

```json
{
  "code": 0,
  "message": "OK",
  "data": {}
}
```

- 通用失败响应：

```json
{
  "code": 1002,
  "message": "参数校验失败",
  "errors": {}
}
```

## 7.2 HTTP 状态码约定

- `200` 成功
- `400` 参数错误
- `401` 未认证 / Token 无效
- `403` 权限不足
- `404` 资源不存在
- `409` 业务冲突
- `500` 服务端异常

## 7.3 业务错误码建议沿用

- `1001` 认证失败
- `1002` 参数校验失败
- `1003` 资源未找到
- `1004` 权限不足
- `1005` 业务冲突

## 7.4 分页约束

- 默认使用 `page` + `size`
- 默认页码：`page=1`
- 默认大小：`size=20`
- 普通列表最大：`size<=100`
- 推荐列表最大：`size<=50`

## 7.5 JWT 约束

登录成功后返回 JWT / token，至少应包含：

- `userId`
- `role`
- 过期时间

当前实现建议：

- 先在服务层和安全组件中定义 Token 生成/解析接口
- 真实 JWT 实现可以在服务框架搭好后补上

## 8.2 需求状态

- `PENDING`
- `REVIEWING`
- `IN_PROGRESS`
- `COMPLETED`
- `CANCELLED`

临时解释：

- `PENDING`：已发布，待接单 / 可展示
- `REVIEWING`：管理后台审核中
- `IN_PROGRESS`：已有人接单，流程进行中
- `COMPLETED`：已完成
- `CANCELLED`：已取消

注：

- Sprint 看板里有“开放中”的表述，当前临时映射为 `PENDING`
- 如果后续前后端统一改成 `OPEN`，需要同步修正文档与枚举

- `ACCEPTED`
- `IN_PROGRESS`
- `COMPLETED`
- `CANCELLED`

补充说明：

- 需求状态与订单状态不是一回事
- 接单成功时：
  - 需求状态应从 `PENDING` 进入 `IN_PROGRESS`
  - 订单状态初始建议为 `ACCEPTED`

## 8.4 通知类型

- `ORDER_ACCEPTED`
- `STATUS_CHANGED`
- `REVIEW_RECEIVED`

## 8.5 评分约束

## 10. 分任务详细安排

## 10.1 USR-03 用户注册/登录/资料管理服务

### 目标

- 注册时完成学号唯一性校验
- 密码加密存储
- 登录成功返回 token
- 支持个人资料读取与更新

### 预计服务接口

建议定义：

- `AuthApplicationService`
  - `register(RegisterCommand command)`
  - `login(LoginCommand command)`
  - `getProfile(Long userId)`
  - `updateProfile(Long userId, UpdateProfileCommand command)`

### 输入输出约束

- 注册入参至少包含：
  - `email`
  - `verificationCode`
  - `studentId`
  - `password`
  - `nickname`
  - `avatarUrl`
- 登录入参：
  - `loginId`
  - `password`
- 输出：
  - `token`
  - `expiresIn`
  - `user summary`

### 业务规则

- `studentId` 必须唯一
- `email` 应视为唯一注册标识之一
- 密码不能明文保存
- 被封禁用户不能登录
- 更新资料只能修改自己的资料

### 测试重点

- 重复学号注册返回冲突
- 密码错误登录失败
- 封禁用户登录失败
- 修改他人资料返回 `403`

### 数据库暂留空

- `TODO-DB-USR-01`：`UserRepository` 的 MyBatis Mapper / XML 暂不实现
- `TODO-DB-USR-02`：邮箱验证码发送与校验存储机制暂不实现

## 10.3 ORD-03 接单/状态流转/订单历史服务

### 目标

- 完成接单
- 完成订单状态流转
- 完成订单历史查询
- 保证同一需求不能重复接单

### 预计服务接口

- `OrderApplicationService`
  - `accept(Long operatorId, Long demandId, AcceptOrderCommand command)`
  - `updateStatus(Long operatorId, Long orderId, UpdateOrderStatusCommand command)`
  - `getDetail(Long operatorId, Long orderId)`
  - `listHistory(Long operatorId, OrderHistoryQuery query)`

### 业务规则

- 接单前必须校验需求状态为 `PENDING`
- 不能给自己发布的需求接单
- 同一需求只能成功接单一次
- 非订单参与方不能查看订单详情
- 合法状态流转建议：
  - `ACCEPTED -> IN_PROGRESS`
  - `IN_PROGRESS -> COMPLETED`
  - `ACCEPTED -> CANCELLED`
- 非法跳转直接返回 `409`

### 并发控制约束

实现目标是“同一需求不能重复接单”，当前先按如下层次设计：

1. 服务层先查状态，发现非 `PENDING` 立即拒绝
2. Repository 接口预留“条件更新 / 乐观锁 / 唯一约束冲突”能力
3. 真实数据库层接入后，再落最终防线

### 测试重点

- 已被接单的需求再次接单失败
- 自己不能接自己的需求
- 非参与方无法查看订单
- 非法状态流转失败

### 数据库暂留空

- `TODO-DB-ORD-01`：`OrderRepository` 真实表结构与 demand/order 分离实现待确认
- `TODO-DB-ORD-02`：重复接单最终以数据库唯一约束还是条件更新保证，待 DAO 层负责人确认
- `TODO-DB-ORD-03`：事务边界与锁策略待接入真实 Mapper 后实现

## 10.4 REV-03 双向评价/信用分重算服务

### 目标

- 支持订单完成后双方各自评价一次
- 评价完成后更新被评价人信用分

### 预计服务接口

- `ReviewApplicationService`
  - `submit(Long operatorId, Long orderId, SubmitReviewCommand command)`
  - `listUserReviews(Long targetUserId, ReviewQuery query)`
  - `recalculateCreditScore(Long userId)`

### 业务规则

- 只有订单参与双方可评价
- 只有订单完成后可评价
- 同一订单同一评价人只能评价一次
- 评分必须在 `1-5`
- 双方评价完成后触发信用分重算

### 信用分临时规则

先采用简单可落地版本：

- 默认初始信用分：`100`
- 重算方式：基于历史评分均值做整数映射或增量更新

说明：

- 具体公式可简化，但必须保证可测试、可解释
- 后续如需改公式，只改 `review` 模块服务，不影响外部接口

### 测试重点

- 未完成订单不能评价
- 重复评价失败
- 评分越界失败
- 双向评价完成后信用分更新逻辑被触发

### 数据库暂留空

- `TODO-DB-REV-01`：评价记录唯一性约束实现待 DAO 层补充
- `TODO-DB-REV-02`：信用分更新是直接写 `sys_user.credit_score` 还是异步汇总，后续再定

## 10.6 MAT-03 简单推荐排序服务

### 目标

- 基于用户历史行为，为需求列表提供简单推荐排序

### 预计服务接口

- `RecommendationApplicationService`
  - `recommendDemands(Long userId, DemandQuery baseQuery)`

### 业务规则

- 推荐只做排序增强，不改变需求列表查询的基础筛选条件
- 推荐开关可配置
- 当前建议规则优先级：
  - 同分类历史偏好
  - 历史接单频次
  - 时间匹配

### 实施策略

- 第一版可以在服务层对查询结果二次排序
- 不强求一开始就写成复杂 SQL
- 后续若数据量增大，再下沉到 DAO / SQL 优化

### 测试重点

- 开关关闭时应退化为普通排序
- 有历史偏好的用户应优先看到高相关分类
