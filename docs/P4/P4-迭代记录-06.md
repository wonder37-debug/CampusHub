说明：

- 当前后台服务先落在应用服务层，不提前引入独立 DAO
- 继续复用现有 `user`、`demand`、`order` 仓储接口，在服务层完成权限与业务编排

## 2. 管理员鉴权规则已落地

本轮已经在后台服务层统一收口管理员鉴权：

- 所有后台入口都要求 `operatorId` 对应用户存在
- 只有 `role=ADMIN` 的用户可以访问后台服务
- 普通用户访问后台服务时统一返回 `403`

这与 P3 文档中“所有 admin 接口必须经过 RBAC 校验”的约束保持一致。

## 3. 用户管理能力已落地

当前后台服务已经支持：

- 用户列表分页查询
- 按关键字检索用户
- 封禁用户
- 解封用户

本轮具体规则：

- 用户列表查询支持按 `nickname`、`email`、`studentId` 模糊匹配
- 管理员不能封禁自己
- 已封禁用户不能重复封禁
- 已激活用户不能重复解封

## 4. 需求审核能力已落地

当前后台服务已经支持：

- 查询待审核需求列表
- 审核需求 `approve / reject`

本轮具体规则：

- 待审核列表只返回 `REVIEWING` 状态需求
- 列表支持关键字筛选与分类筛选
- 只有 `REVIEWING` 状态的需求允许审核
- `approve` 会把需求流转到 `PENDING`
- `reject` 会把需求流转到 `CANCELLED`

说明：

- 这里先沿用当前项目已有 `DemandStatus.REVIEWING` 语义承接审核中状态
- 审核驳回后的最终状态暂用 `CANCELLED` 表达，后续如 DAO/前后端联调需要，也可以再收敛为更细化状态

## 5. 统计看板第一版已落地

当前后台统计接口已经返回以下数据：

- `dailyActiveUsers`
- `totalUsers`
- `totalDemands`
- `pendingReviewDemands`
- `totalOrders`
- `completedOrders`
- `categoryDistribution`

当前统计口径说明：

- `dailyActiveUsers`：按用户 `createdAt` 或 `updatedAt` 是否为当天做轻量统计
- `categoryDistribution`：按当前需求分类聚合
- 当前先返回可测试、可解释、可联调的简化口径，后续接数据库后可进一步精细化

## 6. 为后台统计补齐的仓储能力

为了让后台服务能够做全量聚合，本轮顺手为现有仓储补充了全量查询接口：

- `UserRepository.findAll()`
- `OrderRepository.findAll()`

并同步补齐了内存仓储实现。

说明：

- 这一步仍然是服务层驱动下的接口补充
- 没有引入数据库实现细节，符合当前任务边界

## 按 P3 / P1 吸收的关键约束

### 来自 P3 API 规范

- 管理后台接口必须要求管理员权限
- 用户列表、待审核需求列表都需要支持分页
- 后台能力包含封禁/解封、审核、统计看板
- 后台操作建议保留 `operatorId` 与 `reason`

### 来自 P1 需求

- 管理后台承担社区治理与违规处理职责
- 需求审核与用户管理应具备明确权限边界
