# P4 数据库接口调用规范

## 1. 文档目的

本文档用于和数据库/DAO 层同学对接当前 `backend` 服务层已经稳定下来的仓储接口语义，帮助后续将内存仓储实现替换为 MyBatis / SQL 实现时，尽量做到：

1. 方法签名不乱改
2. 业务语义不跑偏
3. 并发约束与唯一约束不遗漏
4. 服务层与 DAO 层职责边界清晰

说明：

- 本文档面向当前仓库中的仓储接口，不面向 Controller。
- 当前服务层已经通过单元测试验证的规则，以这里的“接口契约”作为 DAO 对接优先依据。
- 如果后续 SQL 设计与本文档存在小冲突，优先保持服务层语义稳定，再小范围调整 Mapper 设计。

## 3. 对接总原则

## 3.1 服务层与 DAO 层职责边界

当前约定如下：

- 服务层负责参数校验、权限判断、状态流转、业务冲突判断。
- DAO 层负责数据读写、唯一约束落地、基础查询性能。
- DAO 层不要重复实现服务层的高层业务判断，但必须兜住关键数据一致性约束。

例子：

- “只有订单参与方可以看订单”属于服务层。
- “同一需求不能重复接单”除了服务层判断外，DAO/数据库层也必须做最终兜底。

## 3.2 Repository 方法通用语义

所有 `save(...)` 方法当前统一语义：

- `id == null`：新增
- `id != null`：更新

所有 `findBy...(...)` 方法当前统一语义：

- 查询不到返回 `Optional.empty()`
- 不在 DAO 层直接抛业务异常

所有 `findAll()` / `findBy...List()` 方法当前统一语义：

- 返回非 null 集合
- 查不到时返回空列表，不返回 null

## 3.3 排序责任

当前大多数排序在服务层完成，因此 DAO 层第一版可以先不把排序固化到 SQL。

建议：

- 第一版 DAO 只保证“查得对”
- 第二版再逐步把高频筛选与排序下沉到 SQL

---

## 4. 错误与异常约定

DAO 层对接时请遵守以下原则：

- 查询不到：返回空对象/空列表，不主动抛业务异常
- 唯一约束冲突：建议抛出底层异常，由服务层或适配层转成 `BusinessException`
- 数据库不可用：保留原始异常，交由全局异常处理

当前服务层使用的业务错误码见：

- [ErrorCode.java](/E:/homeworki/sec-ii-2026/backend/src/main/java/com/campushub/backend/common/exception/ErrorCode.java)

核心错误码语义：

- `1001 AUTH_FAILED`：认证失败
- `1002 VALIDATION_FAILED`：参数/格式错误
- `1003 RESOURCE_NOT_FOUND`：资源不存在
- `1004 PERMISSION_DENIED`：权限不足
- `1005 BUSINESS_CONFLICT`：业务冲突

DAO 层不要求直接感知这些错误码，但要知道哪些数据库约束会被映射成这些业务语义。

## 5. 各仓储接口契约

## 5.1 UserRepository

接口方法：

- `Optional<User> findById(Long id)`
- `Optional<User> findByStudentId(String studentId)`
- `Optional<User> findByEmail(String email)`
- `Optional<User> findByLoginId(String loginId)`
- `List<User> findAll()`
- `User save(User user)`

DAO 实现要求：

- `studentId` 必须唯一
- `email` 必须唯一
- `findByLoginId` 需要支持“学号或邮箱”二选一匹配
- `findAll` 供后台列表和统计使用

建议 SQL / 索引：

- `student_id` 唯一索引
- `email` 唯一索引
- `role`、`status` 普通索引可按需要补充

注意事项：

- 当前服务层会频繁读取 `role`、`status`、`creditScore`
- 不要在 DAO 层自动过滤封禁用户，是否可操作由服务层判断

## 5.2 DemandRepository

接口方法：

- `Optional<Demand> findById(Long demandId)`
- `List<Demand> findAll()`
- `Demand save(Demand demand)`

DAO 实现要求：

- 需求状态字段必须完整支持：
  - `PENDING`
  - `REVIEWING`
  - `IN_PROGRESS`
  - `COMPLETED`
  - `CANCELLED`
- `save` 需要完整持久化匿名标识、时间窗、奖励、标签等字段

建议 SQL / 索引：

- `publisher_id`
- `status`
- `category`
- `campus_zone`
- `created_at`

注意事项：

- 当前列表筛选、推荐筛选、后台审核列表都复用 `findAll()` 结果
- 后续如果下沉查询到 SQL，建议新增更细分接口，不要直接改变 `findAll()` 语义

## 5.3 OrderRepository

接口方法：

- `Optional<Order> findById(Long orderId)`
- `Optional<Order> findByDemandId(Long demandId)`
- `List<Order> findByParticipant(Long userId)`
- `List<Order> findAll()`
- `Order save(Order order)`

DAO 实现要求：

- 同一需求只能成功创建一条有效订单
- `findByParticipant(userId)` 需要同时覆盖：
  - 发布者 `publisherId`
  - 接单者 `accepterId`

强约束：

- `demand_id` 必须唯一，或通过等价机制保证不会重复接单

建议 SQL / 索引：

- `demand_id` 唯一索引
- `publisher_id`
- `accepter_id`
- `status`
- `created_at`

并发说明：

- 当前服务层已经先查 `findByDemandId` 再保存
- 但最终防重必须靠数据库唯一约束或原子条件更新兜底
- 如果是插入冲突，请让异常可以被服务层识别成“已被接单”

## 5.4 ReviewRepository

接口方法：

- `Optional<Review> findByOrderIdAndAuthorId(Long orderId, Long authorId)`
- `List<Review> findByTargetId(Long targetId)`
- `List<Review> findByOrderId(Long orderId)`
- `Review save(Review review)`

DAO 实现要求：

- 同一订单、同一评价作者，只允许一条评价
- `findByTargetId` 供信用分重算和评价列表展示复用

强约束：

- `(order_id, author_id)` 组合唯一

建议 SQL / 索引：

- `(order_id, author_id)` 唯一索引
- `target_id`
- `created_at`

注意事项：

- 当前信用分重算依赖 `findByTargetId`
- 不要在 DAO 层做评分范围校验，评分范围由服务层控制

## 5.5 NotificationRepository

接口方法：

- `Optional<Notification> findById(Long notificationId)`
- `List<Notification> findByUserId(Long userId)`
- `Notification save(Notification notification)`

DAO 实现要求：

- `findByUserId` 返回该用户收到的全部通知
- 已读标记需要支持更新

建议 SQL / 索引：

- `user_id`
- `type`
- `is_read`
- `created_at`

注意事项：

- 排序当前由服务层完成
- 后续如果通知量变大，可以新增“按用户分页查询”的专用接口，不建议直接改变当前接口语义

## 7. 建议的 Mapper 对接方式

为了减少和服务层的摩擦，建议数据库同学按下面方式接：

1. 先严格按当前 `Repository` 接口补 MyBatis 实现，不改服务层签名
2. 单元测试先让原有服务层测试全部通过
3. 性能不足时，再新增更细粒度查询接口
4. 新增接口时先和服务层负责人确认命名与语义，再下沉到 SQL

不建议：

- 直接把服务层中的筛选逻辑偷偷挪进现有 `findAll()`
- 为了省事把“查不到”改成抛异常
- 修改已有方法返回语义去贴合某一条 SQL

---

## 8. 推荐对接优先级

如果时间有限，建议数据库层按这个顺序对接：

1. `UserRepository`
2. `DemandRepository`
3. `OrderRepository`
4. `ReviewRepository`
5. `NotificationRepository`
6. 推荐与后台统计专项 SQL

原因：

- 用户、需求、订单是主链路
- 评价、通知依赖订单闭环
- 推荐与后台统计当前已有服务层兜底，可以稍后优化

---

## 9. 对接完成标准

数据库/DAO 对接完成后，建议至少满足以下验收条件：

1. 不修改现有服务层公开接口
2. `mvn test` 全量通过
3. 重复学号/邮箱不能写入
4. 同一需求不能重复接单
5. 同一订单同一作者不能重复评价
6. 后台列表、推荐、通知、历史订单查询结果与当前内存实现语义一致

---

## 10. 备注

这份规范的核心目标不是把 SQL 细节定死，而是把“服务层已经稳定依赖的仓储语义”定下来。只要数据库层实现始终围绕这个目标推进，后续无论是前后端联调，还是继续补 Controller、推荐优化、后台统计增强，都会顺很多。
