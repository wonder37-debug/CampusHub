# CampusHub 问题报告

> **文档说明**：本报告供团队成员及 AI 辅助开发工具参考。问题按优先级分为三级：
>
> - **P0**：严重 Bug，影响核心功能或存在数据安全风险，必须优先修复
> - **P1**：重要功能缺陷，影响用户正常使用，需尽快修复
> - **P2**：交互细节与 UI 优化，不影响核心功能，迭代修复

------

## P0 — 严重缺陷（Critical Bug）

------

### P0-01 · 过期需求无自动下架机制，仍可被接单

**页面**：`/demands`（需求列表首页）

**问题描述**：当需求的 `endTime` 已过期，系统存在以下三个问题：

1. 过期需求仍在首页正常展示，状态仍显示为"开放中"（`PENDING`），不会自动转为"已过期"
2. 前后端均未对过期时间做接单校验，用户仍可对过期需求成功接单
3. 过期需求与正常需求混在一起，严重影响用户体验和平台信任度

**修改建议**：

- 后端新增定时任务（如每5分钟执行一次），自动将 `endTime < now` 且状态为 `PENDING` 的需求状态更新为 `EXPIRED`
- 前端需求列表默认过滤掉 `EXPIRED` 状态
- 接单接口后端增加过期校验，若需求已过期则返回 `409` 并提示"该需求已过期，无法接单"

------

### P0-02 · 信用分计算公式异常，单次差评导致分数骤降

**页面**：评价提交后 → 用户信用分更新

**问题描述**：当前信用分计算公式存在严重问题，一次差评（1星）可直接将用户信用分从 100 分降至 20 分，跌幅高达 80 分，不符合评分平滑原则，易被恶意差评利用。

**修改建议**：

- 采用加权移动平均算法：新信用分 = 旧信用分 × 0.9 + 新评分归一化值 × 0.1（信用分变动幅度每次不超过 5~10 分）
- 或改为基于历史评价均值计算：信用分 = 60 + (`rating_sum` / `review_count`) × 8，满分 100
- 设置单次评价对信用分影响上限，防止恶意刷分

------

### P0-03 · 管理后台统计数据全部显示为 0，数据接口异常

**页面**：`/admin`（管理后台看板）

**问题描述**：管理后台的"今日活跃用户"、"用户总数"、"待审需求"、"已完成订单"四项统计数据均显示为 0，与实际数据不符，数据接口存在异常。

**修改建议**：

- 排查后端 `/api/v1/admin/stats` 接口返回数据，确认 `dau`、`ordersCount`、`demandsCount` 字段是否正确赋值
- 检查 SQL 查询逻辑，日活统计应基于今日有登录记录或操作记录的用户数
- 前端确认接口响应后数据绑定是否正确

------

### P0-04 · 推荐排序功能未生效，"加载更多"状态永久显示

**页面**：`/demands`（需求列表首页）

**问题描述**：

1. 选择"推荐排序"后，列表展示顺序与后端推荐算法结果不一致，推荐排序实际未生效
2. 点击"刷新列表"后，页面出现"正在加载更多..."字样，且该状态永久不消失，列表无法正常刷新

**修改建议**：

- 检查前端调用推荐接口 `/api/v1/recommendations` 时的参数传递是否正确，确认返回结果按 `rank` 字段排序渲染
- 修复"刷新列表"的加载状态管理，请求完成后（成功或失败）必须重置 loading 状态
- 将"推荐排序"设为默认排序方式

------

### P0-05 · 消息通知点击后无法跳转到关联订单或需求

**页面**：`/notifications`（消息页面）

**问题描述**：消息列表中每条消息均有"点击可查看相关订单或需求详情"的引导文字，但实际点击任意消息后无任何跳转响应，功能完全失效。

**修改建议**：

- 每条通知记录中已有 

  ```
  related_id
  ```

   字段，点击消息后应根据 

  ```
  type
  ```

   字段判断跳转目标：

  - `ORDER_ACCEPTED` / `STATUS_CHANGED` → 跳转至 `/demands/{related_id}`
  - `REVIEW_RECEIVED` → 跳转至对应订单详情页

- 同时修复：切换页面后消息已读状态未持久化的问题（需调用后端标记已读接口，而非仅在前端维护状态）

------

### P0-06 · 发布需求时报酬无余额校验，可填写超出余额的金额

**页面**：`/demands/new`（发布新需求）

**问题描述**：用户在发布需求时填写的报酬金额无上限校验，即使填写的报酬金额超出当前账户余额，系统也允许发布，导致后续接单结算时无法正常扣款。

**修改建议**：

- 前端：在报酬输入框失焦时实时校验，若超出当前用户 `balance` 则显示红色错误提示"报酬不能超过当前可用余额 ¥XX"
- 后端：发布需求接口增加余额校验，不足时返回 `400` 错误

## P1 — 重要功能缺陷（High Priority）

------

### P1-01 · 未登录状态下，入口引导缺失，用户无法点击跳转登录

**页面**：全局导航栏、`/orders`、`/notifications`

**问题描述**：

1. 导航栏右上角显示"未登录访客"，但点击无任何响应，用户无法通过此处进入登录/注册页
2. 订单页和消息页提示"请先登录查看..."，同样无法点击跳转，缺乏引导

**修改建议**：

- 将"未登录访客"改为可点击的登录入口按钮，点击后跳转至 `/auth`
- "请先登录查看..."改为带下划线的可点击链接文字，点击跳转至 `/auth`

------

### P1-02 · 登录/注册页密码输入框缺少显示/隐藏密码功能

**页面**：`/auth`（登录与注册页）

**问题描述**：密码输入框始终为密文状态，缺少切换明文/密文的"小眼睛"图标，用户无法确认自己的输入内容。

**修改建议**：

- 在密码输入框右侧添加眼睛图标，点击切换 `input type` 在 `password` 和 `text` 之间切换
- 对所有密码相关输入框（登录密码、注册密码、确认密码）统一添加该功能

------

### P1-03 · 部分错误提示语为英文，未做国际化处理

**页面**：注册页、登录页及全局错误提示

**问题描述**：多处错误提示直接暴露英文后端消息，包括但不限于：

- `password must be at least 8 characters`
- `verification code is invalid`
- `verificationCode must not be blank`
- `studentId already registered`
- `user has been banned`
- `order is already in target status`（订单详情页底部）

**修改建议**：在前端统一建立错误码映射表，将后端返回的英文错误信息转换为对用户友好的中文提示，例如：

- `studentId already registered` → "该学号已被注册，请直接登录"
- `user has been banned` → "您的账号已被封禁，请联系管理员"
- `order is already in target status` → "订单状态已更新，请刷新页面"

------

### P1-04 · 管理员角色未与普通用户隔离，权限边界混乱

**页面**：管理员账号登录后的全局导航

**问题描述**：

1. 管理员登录后导航栏仍显示"订单"入口，可参与普通用户的发单、接单流程，角色定位错误
2. 缺少管理员账号的增删渠道（如何晋升/撤销管理员权限），目前只能手动修改数据库

**修改建议**：

- 管理员登录后，导航栏隐藏"订单"入口，将"我的"替换为"管理后台"
- 管理员账号不允许发布需求和接单，相关接口后端增加角色校验
- 在超级管理员后台新增"角色管理"功能，支持将普通用户提升/降级为管理员

------

### P1-05 · 审核中的需求出现在首页大厅，未通过审核的订单对外可见

**页面**：`/demands`（需求列表）

**问题描述**：状态为 `REVIEWING`（审核中）的需求出现在首页大厅列表，用户可以查看并尝试接单，违反了"只有审核通过后才公开"的业务逻辑。

**修改建议**：

- 需求列表接口默认只返回 `status = PENDING`（审核通过，开放接单）的需求
- `REVIEWING` 状态的需求只对发布者本人和管理员可见
- 首页大厅状态筛选器默认选中"开放中"

------

### P1-06 · 需求详情页接单区域逻辑混乱，状态提示不清晰

**页面**：`/demands/{demandId}`

**问题描述**：

1. 无论订单处于何种状态，"当前需求暂时不可接单"文字始终展示，未根据实际不可接单原因区分
2. "开始执行"按钮在发布者视角也显示，该按钮应仅对接单人展示
3. 接单留言区域对所有人展示，包括第三方用户，隐私保护缺失
4. 接单留言缺少确认/提交按钮，无法单独发送留言

**修改建议**：

- 根据不可接单原因展示明确提示：
  - 已被他人接单 → "该需求已被接单，来晚了一步"
  - 是自己发布的需求 → "这是您自己发布的需求，不能自行接单"
  - 订单已完成/取消 → "该需求已结束"
- "开始执行"按钮仅在接单人视角且订单状态为 `ACCEPTED` 时展示
- 接单留言区仅对未登录用户/第三方隐藏；对接单人展示输入框并增加"确认留言"按钮

------

### P1-07 · 订单详情页双向评价显示不完整，评价区域角色未区分

**页面**：`/demands/{demandId}`（已完成订单）

**问题描述**：

1. "相关评价"板块只显示一方评价，无法看到双向的发布者↔接单人互评记录
2. 评分显示不统一，部分显示"分"，部分显示"星"
3. 已提交过评价的用户仍显示评价提交表单，无"已评价"的明确提示
4. "已确认完成，等待对方确认"这条描述在发布者和接单人视角下应显示不同内容

**修改建议**：

- 在"相关评价"区域同时展示双方评价，格式为：`发布者 → 接单人：[评价内容] [评分]` 和 `接单人 → 发布者：[评价内容] [评分]`
- 评分统一改为"X 星"或"X 分"，全局统一
- 用户已提交评价后，隐藏评价表单，改为展示"您已提交评价，不可重复评价"
- 接单人视角显示"已确认完成，等待发布者确认"；发布者视角显示"接单人已确认完成，等待您的确认"

------

### P1-08 · 保存个人资料无操作反馈，昵称未做唯一性校验

**页面**：`/profile`（个人中心）

**问题描述**：

1. 点击"保存资料"后，无任何成功或失败的提示（Toast/弹窗），用户不知道操作是否成功
2. 昵称可设置为与其他用户相同的值，未做唯一性约束，导致平台内出现同名用户

**修改建议**：

- 保存成功后展示绿色 Toast 提示"资料已更新"，失败时展示红色提示及具体原因
- 昵称修改时后端校验唯一性，若重复则返回错误提示"该昵称已被使用，请更换"

------

### P1-09 · 需求发布后未通知管理员，管理员消息无法跳转审核界面

**页面**：`/notifications`（管理员消息页）

**问题描述**：

1. 用户发布需求后，管理员未收到新需求待审核的站内通知
2. 即便有消息，管理员点击消息后无法直接跳转至对应需求的审核界面

**修改建议**：

- 需求创建成功后，后端向所有 `role = ADMIN` 的用户发送通知，类型为 `REVIEW_REQUEST`，`related_id` 指向需求 ID
- 管理员点击该类型消息后，跳转至 `/admin?tab=review&demandId={id}` 直接定位到该需求的审核操作

------

### P1-10 · 我的订单页面信息不完整，状态标签使用英文

**页面**：`/orders`（我的订单）及订单详情

**问题描述**：

1. 从"我的订单"进入订单详情后，信息展示不完整，缺少需求的分类、地点、时间、报酬等核心字段
2. "操作"标签悬浮在角落，语义不明，不知道点击会触发什么
3. "已完成"状态标识字号过小，不够醒目

**修改建议**：

- 订单详情页展示完整需求信息：标题、分类、地点、开始/结束时间、报酬、发布者信息
- "操作"文字替换为具体操作按钮（如"查看需求详情"、"提交评价"）
- "已完成"标签使用更大字号或加粗绿色样式

## P2 — UI 优化与体验改善（Medium Priority）

------

### P2-01 · 个人中心页面排版混乱，字段标注不清晰

**页面**：`/profile`

**问题描述**：

1. 学号重复展示两次（个人信息区 + 编辑区），令人困惑
2. 角色显示"学生"、状态显示"正常"，过于简略，语义不够完整
3. 整体排版偏旧，视觉层次不够清晰

**修改建议**：

- 学号只展示一次，格式改为"学号：241880520"
- 角色改为"当前身份：学生用户"；状态改为"当前状态：正常"
- 参考现代化个人中心设计，将信息分组展示（基本信息、账户信息、操作按钮）

------

### P2-02 · 管理后台看板排版不美观，功能入口层次不清

**页面**：`/admin`

**问题描述**：

1. "管理中心"标题字号过小，且下方说明文字莫名换行，视觉效果差
2. 统计卡片（今日活跃用户、用户总数等）不可点击，无法下钻查看详情
3. "待审核需求"和"平台用户列表"两个模块并列排放，空间拥挤

**修改建议**：

- 加大"管理中心"标题字号，修复说明文字换行问题
- 统计卡片增加点击跳转功能（如点击"今日活跃用户"跳转至筛选了今日活跃用户的用户列表）
- 将审核队列和用户列表改为上下分区排列，或通过 Tab 切换

------

### P2-03 · 审核队列功能不完善，缺少拒绝理由填写

**页面**：`/admin`（审核队列模块）

**问题描述**：

1. 审核队列中每条需求不可点击查看详情
2. 搜索框包含"地点"字段，与审核场景不相关
3. 搜索框右侧缺少确认按钮（如放大镜图标）
4. 缺少按校区筛选的下拉选项
5. 点击"拒绝"后未弹出理由填写框，发布者无法得知被拒原因

**修改建议**：

- 每条审核需求卡片可点击，点击后打开需求详情弹窗或跳转至详情页
- 搜索框只保留"标题"搜索，去掉地点，并在右侧添加放大镜图标作为确认按钮
- 新增"校区"筛选下拉框
- 点击"拒绝"后弹出对话框，要求填写拒绝理由（必填），提交后通过站内信发送给发布者

------

### P2-04 · 平台用户列表排版错乱，缺少排序和筛选功能

**页面**：`/admin`（用户列表模块）

**问题描述**：

1. 用户列表排版错乱，昵称有时横排、有时竖排，与其他字段交错重叠
2. 搜索框未区分搜索字段（学号/邮箱/昵称混合搜索体验差）
3. 缺少列排序功能（信用分升降序等）
4. 角色列和状态列缺少筛选功能

**修改建议**：

- 修复表格布局，确保各列等宽固定，昵称列设置最大宽度并截断显示
- 搜索框改为下拉选择搜索类型（学号 / 邮箱 / 昵称）再输入关键词
- 信用分列添加升降序排序按钮
- 角色列和状态列标题改为可点击的筛选标签（全部 / 学生 / 管理员；全部 / 正常 / 已封禁）

------

### P2-05 · 首页大厅排版与交互细节问题

**页面**：`/demands`

**问题描述**：

1. "下拉刷新"和"刷新列表"功能重复，且"刷新列表"按钮与左侧筛选项不对齐
2. 缺少标题搜索框
3. 排序选项中存在"距离最近"（项目未接入地理位置功能），以及名称不直观的"时间倒序"、"报酬从高到低"

**修改建议**：

- 删除"下拉刷新"入口，保留"刷新列表"并对齐排版
- 在筛选区新增搜索框，支持按需求标题搜索
- 删除"距离最近"排序选项
- 排序选项重命名：时间倒序 → 时间最近；报酬从高到低 → 报酬最高
- 默认排序改为"推荐排序"

### P2-06 · 需求卡片信息展示缺少字段标识

**页面**：`/demands`（需求列表卡片）及需求预览

**问题描述**：

1. 地点、描述、发布者直接显示内容，缺少字段标识，语义不清
2. 报酬为 0 时显示"一杯奶茶"，视觉上不够专业，且与信息规范性不符
3. 信用分全部显示 0.0 分，数据未正确加载

- `娃哈哈` → `发布者：娃哈哈`
- 报酬为 0 时直接显示 `¥0`，不使用文字替代
- 修复信用分显示，确保正确从后端读取 `credit_score` 字段

### P2-07 · 发布需求按钮过小，入口不醒目
