# **ADR记录:**

## **ADR-001:采用微服务架构**

### **状态**

已废弃

### **背景**

在系统设计初期,考虑到校园互助系统未来可能面临的功能扩展(如增加跑腿、二手交易、技能交换等独立模块)以及潜在的高并发场景(如特定时间段的失物招领抢单),团队考虑参考 AI 的建议采用基于 Spring Cloud 的微服务架构,以实现各业务模块的物理隔离和独立演进。

### **决策**

决策 我们决定采用基于 Spring Cloud Alibaba 体系的微服务架构。具体包含:

* 网关层:使用 Spring Cloud Gateway,集成JWT 实现无状态鉴权。  
* 服务层:按照 DDD(领域驱动设计)原则拆分,各服务拥有独立数据库 Schema。  
* 异步处理:引入 RabbitMQ 解耦帖子发布后的通知推送逻辑。

### **理由**

* 解耦性:微服务允许不同模块使用最适合的技术栈。  
* 可扩展性:支持针对特定热点服务(如任务中心)进行水平扩容,以应对应对未来可能加入的 二手交易平台和外卖跑腿 等异构业务。  
* 故障隔离:单个服务的崩溃不会导致整个系统不可用。  
* 团队分工:为了适配4人团队的并行开发,微服务可以强制从物理上隔离代码冲突,避免 Monolith 模式下的频繁 Merge 冲突。

### **后果**

* 正面影响:系统架构具备极高的灵活性和工业级支撑能力,可轻松应对高并发场景。  
* 负面影响:大幅增加了运维复杂度(监控、链路追踪);引入了分布式事务的一致性难题。  
* 需要关注的风险:开发工作量超出了10周的项目周期限制;团队在处理容器编排和微服务治理方面的经验不足。

**AI 辅助记录**

**AI 初稿内容摘要**

* 核心建议:推荐采用“前后端分离的微服务架构”,认为这能应对高并发和多端接入的需求。  
* 技术栈推荐:建议使用 Spring Cloud Gateway 作为网关,RabbitMQ 处理异步通知,并使用Docker \+ K8s进行容器化部署。  
* 业务逻辑:建议通过“用户中心”、“任务中心”、“消息中心”等微服务实现功能拆分。

**人工修订内容:**

1. 加上“应对未来可能加入的二手交易平台 和 外卖跑腿 等异构业务”  
2. 决策部分由“我们最初计划将系统拆分为以下独立服务:  
   1. 用户服务:负责身份认证与信用评价。  
   2. 任务/互助服务:负责帖子发布与订单流转。  
   3. 消息/即时通讯服务:负责用户间沟通。  
      网关服务:统一路由与鉴权。”改到现在  
3. 理由添加团队分工

**修订理由:**

1. AI 给的初稿太通用了。考虑到校园项目的特殊性,后面很可能会加一些奇奇怪怪的功能,把这些具体场景写进去,能让这个决策更有针对性。  
2. 只说拆分服务太虚了,没法落地。具体的选型需要把它落实到文档里  
3. 这一点 AI没考虑到。微服务对这种小组协作最大的好处其实是“互不干扰”。防止代码冲突

## **ADR-002:选择 MySQL 作为核心业务数据库**

### **状态**

已接受

### **背景**

我们需要存储用户信息、任务编号、积分交易和评价数据。这些数据具有强关联性(例如积分交易必须严格绑定发起者、接收者与对应任务编号),且在交易环节需要保证数据的一致性(ACID 特性)。团队只有4人,且服务器资源有限。

### **决策**

我们决定采用**MySQL 8.0** 作为主数据库。

### **理由**

* **数据一致性**: MySQL 的 InnoDB 引擎能完美支持复杂事务,确保积分在“任务发布-承接-完成”整个链路中的准确性。  
* **团队熟练度**:团队成员在课程中均学习过SQL,学习成本最低,能保证 10周内完成。  
* **社区支持**:资料丰富,遇到Bug 容易通过AI或社区解决。  
* **资源占用**:相比于复杂的微服务数据库集群,单机 MySQL占用内存较小,适合我们的低配服务器。

### **后果**

* **正面影响**:开发速度快,数据结构严谨,方便进行多表关联查询(如查询某个用户的所有历史任务)。  
* **负面影响**:面对超大规模高并发时,扩展性不如 NoSQL 数据库。  
* **需要关注的风险**:需要做好数据库备份,防止因服务器故障导致的核心交易数据丢失。

**AI 辅助记录**

**AI 初稿内容摘要**:AI 建议使用MySQL,理由是它是工业标准、性能稳定且支持事务。

**人工修订内容**:增加了“团队成员熟练度”和“服务器资源限制”这两条具体约束;将“高性能”修正为“适合我们的低配服务器”;将“商品交易”修改为“积分交易”。

**修订理由**:AI默认推荐是基于通用大型项目的,没有考虑到我们作为学生团队在10周内的交付压力,真实的硬件限制以及团队成员已掌握的知识。AI 默认校园互助平台会进行二手商品的交易,没有考虑我们项目是以任务为形式的平台,并不进行商品的售卖。

## **ADR-003:\[采用模块化单体架构(Modular Monolith)+三层分层\]**

### **状态**

已接受(2026-04-27)

### **背景**

CampusHub 项目在体系结构设计阶段经历了两次决策考虑:

1. 最初考虑纯单体分层架构,但担心后期代码边界崩溃,毕竟纯单体容易变成屎山代码。  
2. 随后尝试微服务架构(ADR-002),因开发效率、学习成本、环境资源等问题不予考虑。

当前需要一种既能快速交付 MVP,又能保持模块边界清晰、未来可拆分的技术方案。团队由4人组成,开发经验较少,开发周期剩余约6周(含编码与测试),核心功能为PO(用户、需求、订单),非功能要求包括页面加载\<2秒、代码可维护、支持后续扩展(智能推荐、WebSocket)。

### **决策**

我们决定采用模块化单体架构(Modular Monolith),并在每个模块内部遵循三层分层(Controller \- Service \- DAO)。

**具体含义:**

* **模块化单体**:整个后端打包为一个可执行Jar,但内部按业务模块(user、demand、order、evaluation、message)进行物理隔离(Maven 多模块或Java 包边界)。模块之间仅通过公开接口(API层)进行调用,禁止直接引用其他模块的内部实现类。  
* **三层分层**:每个模块内部再划分为 Controller、Service、DAO、遵循单向依赖(Controller → Service → DAO),不能跨模块直接调DAO。

### **理由**

1. **兼顾开发效率与边界控制**  
   * 单体打包,无需处理服务发现、分布式事务,调试和部署与普通单体一样简单。  
   * 强制模块化(如Maven 多模块)避免了“大泥球”——编译器会阻止跨模块的非法依赖。  
   * 对比 ADR-001的“普通分层单体”:普通单体只有逻辑分层,无模块边界,赶工时容易互相渗透;模块化单体增加了物理隔离,代价很小。  
2. **为未来拆分微服务预留低成本路径**  
   * 每个模块已有明确的api包和内部实现,未来只需将模块独立部署、加上网络通信层(如Feign),即可逐步演进为微服务。  
   * 智能推荐、即时通讯等P2功能可以作为新模块加入,不会破坏现有结构。  
3. **适合4人团队并行开发**  
   * 每人可负责1-2个模块,模块间接口定义清楚后并行开发,互不阻塞。  
   * 对比微服务:不需要每人启动整套分布式环境;对比普通单体:不会频繁因代码合并冲突而痛苦(因为文件按模块目录隔离)。  
4. **满足质量属性**  
   * 性能:进程内调用,无序列化和网络开销,轻松满足\<2秒。  
   * 可维护性:模块边界+分层,代码位置明确,单元测试可聚焦单个模块。  
   * 可扩展性:同理由2。  
5. **学习成本低**  
   * 模块化单体仍是单体,团队已熟悉Spring Boot 基础,只需学习如何拆分 Maven 模块(约1小时)。微服务的学习曲线(服务发现、网关、分布式配置),成本可忽略。

### **后果**

* **正面影响**:  
  * 代码边界强健,防止了“为了省事直接跨模块调用私有类”的坏习惯。  
  * 构建速度仍然很快(只打一个Jar), CI/CD简单。  
  * 团队开发过程中,合并冲突显著减少(因为不同模块在不同目录)。  
* **负面影响**:  
  * 首次搭建 Maven 多模块项目比单模块略微繁琐(需要定义父pom、模块间依赖版本管理),以及各版本依赖的冲突问题。  
  * 模块之间的数据交互需要编写接口类,不能像普通单体那样直接复用对方的实体。  
  * 如果模块划分不合理,后期调整模块边界需要重构。  
* **需要关注的风险**:  
  * **循环依赖风险**:模块A调模块B,模块B又调模块A。  
    * 应对:使用 Maven的依赖管理禁止循环依赖(maven-enforcer-plugin),或在架构审查中发现后及时修正。  
  * **事务跨模块**:一个业务操作(如下单后扣减用户信用分)需要调用 order 模块和 user 模块。在单体中仍可由上层 Service 编排,使用@Transactional 注解正常生效(因为是同一个数据库连接)。不会像微服务那样需要分布式事务。

**AI 辅助记录**

* **AI 初稿内容摘要**:AI给出了基础模板,包括采用的架构及其具体内容,选择的理由。但未区分模块化单体与普通单体的关键差异,也未提及模块间接口定义的具体方式。AI倾向于只写“优点”,缺少对“负面”影响的分析。  
* **人工修订内容**:在“背景”中明确引用两次架构决策的教训,促使AI去对比分析;补充了模块化单体架构+三层分层是什么的介绍;加入了操作该决策的一些负面效果,以及学习时需要关注的风险;补充“模块化单体”与“普通分层单体”的对比说明,人有时候会混淆二者区别,强调物理隔离的重要性。  
* **修订理由**:AI给出的文档更像通用模板,没有针对 CampusHub的两次历史决策进行定制,没有根据团队成员的实际水平来进行说明介绍,也没有识别出“模块化单体是微服务失败后的自然落地方案”。人工需要把真实的工程上下文(已废弃的ADR-001)融入决策理由中,体现从错误中学习的过程。此外,AI容易低估多模块构建的初期成本和人遵守边界的重要性,人工必须明确写出缓解措施。
