# 架构候选方案对比表

| **对比维度**      | **方案 A：传统分层单体架构 (Layered Monolith)**              | **方案 B：模块化单体架构 (Modular Monolith)**                |
| ----------------- | ------------------------------------------------------------ | ------------------------------------------------------------ |
| **开发复杂度**    | 极低。代码结构扁平，适合初学者快速堆砌功能，无需前期设计复杂的包边界与依赖规则 。 | 中等。需要在项目初期投入一至两天时间设计模块边界、定义模块间数据传输对象 (DTO) 及接口规范 。 |
| **可维护性**      | 较差。代码逻辑容易横向耦合，例如需求模块直接调用用户表的 Mapper，长期迭代会导致排错困难 。 | 优异。业务逻辑高度内聚于各自模块内，边界清晰。修改订单结算逻辑绝不会意外破坏用户身份验证逻辑 。 |
| **可扩展性**      | 较差。系统只能作为一个整体进行扩展，若后续需要引入复杂的推荐算法模块，难以实现资源隔离 。 | 良好。在单体内实现了逻辑隔离，若未来项目获奖并需投入真实校园运营，可低成本将高负载模块剥离为独立服务 。 |
| **性能**          | 极高。纯内存方法调用，没有任何网络反序列化开销，核心页面加载时间必定低于 5 秒 。 | 极高。同样基于单一进程内的内存调用，且因为模块间解耦，更容易针对特定模块引入本地缓存机制提升性能 。 |
| **团队学习成本**  | 最低。基本上手即用 。                                        | 适中。需要团队成员理解“高内聚低耦合”的实践方案，学习控制反转 (IoC) 在模块间通信的应用，但不涉及外部中间件 。 |
| **10 周内可行性** | 极高。能够保证基本盘功能的快速交付 。                        | 高。前期架构设计虽多占部分工时，但在第 6-7 周的核心编码阶段 (Sprint 1)，将极大降低团队代码合并冲突的概率 。 |
