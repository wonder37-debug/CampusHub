## **1\. 系统级用例图 (PlantUML)**

## **2\. 核心用例描述 (Detailed Use Case Specifications)**

### **UC04: 发布互助需求**

* **角色：** 普通学生  
* **前置条件：** 用户已通过 CAS 认证，且“校邻币”余额足以支付所设悬赏。  
* **基本流：**  
  1. 用户在“任务大厅”点击发布按钮，选择具体分类（如：跑腿、咨询、组队）。  
  2. 用户填写任务标题、正文（可选关联课程代码）、截止时间及校区属性。  
  3. 用户设置“校邻币”金额，并选择是否开启“匿名发布”。  
  4. 系统同步调用 **UC18 (违规拦截)**，对内容进行敏感词扫描。  
  5. 扫描通过后，系统冻结用户账户中对应的校邻币金额。  
  6. 任务正式在对应校区的大厅上线。  
* **替代流：**  
  * 4a. 命现代课/代考关键词：系统直接阻止发布，弹出“严禁违规互助”提示，并记录该用户违规行为 1 次。  
  * 5a. 余额不足：系统提示积分获取攻略（引导其去接单）。  
* **后置条件：** 任务状态设为“待领取”，积分进入担保账户。

### **UC08: 拍照存证申请结算**

* **角色：** 普通学生（接单人、发布者）  
* **前置条件：** 任务已被领取且处于“执行中”状态。  
* **基本流：**  
  1. 接单人完成互助（如：将快递送达），在 App 内选择“申请结算”。  
  2. **强制动作**：系统调起摄像头，接单人必须拍摄 1-3 张能够证明任务完成的照片。  
  3. 接单人确认上传后，系统向发布者推送核验提醒。  
  4. 发布者查看照片，确认无误后点击“确认完成”。  
  5. 系统将冻结的积分转入接单人账户，双方校园值（信誉）各增加 1 分。  
* **替代流：**  
  * 4a. 24小时静默结算：若发布者在收到提醒后 24 小时内未点击确认且未发起投诉，系统根据存证照片自动划转积分。  
  * 4b. 拒绝确认：若发布者发起投诉，认为任务未完成，则任务转入 **UC15 (争议仲裁)** 流程。  
* **后置条件：** 任务变为“已归档”，积分完成流转。

### **UC15: 争议仲裁投票**

* **角色：** 学生志愿者管理员  
* **前置条件：** 互助双方产生纠纷，且任务被挂起进入申诉状态。  
* **基本流：**  
  1. 系统根据“校园值”权重，随机抽取 5 名在线的高信用志愿者。  
  2. 志愿者在“仲裁中心”查阅该任务的所有数据：任务描述、聊天记录、**UC08 强制上传的存证照片**。  
  3. 志愿者基于证据独立进行匿名投票（支持发布者 / 支持接单者）。  
  4. 系统根据少数服从多数原则（如 3:2 或以上）自动判定积分归属。  
* **后置条件：** 被判定违规方将根据情节扣除 5-20 分校园值；投票志愿者获得微量积分奖励。
