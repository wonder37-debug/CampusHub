# P3 任务1 - SOLID 检查清单

本文档用于记录 `ai-初版类图.mmd` 中的设计问题，以及修正到 `核心类图-修正版.mmd` 的过程。

## 审查对象

- AI 原始类图：[ai-初版类图.mmd](/E:/homeworki/sec-ii-2026/docs/p3/ai-初版类图.mmd)
- 修正版类图：[核心类图-修正版.mmd](/E:/homeworki/sec-ii-2026/docs/p3/核心类图-修正版.mmd)

## 结论摘要

- AI 原始设计共识别出 `5` 处典型 SOLID 违规，五条原则各命中至少一处。
- 最严重的问题是把发布、接单、结算、推荐、风控、通知、仲裁全部堆进 `CampusHubSystem`，形成明显的“上帝类”。
- 修正后采用与 P2 一致的模块化单体边界，将职责拆入 `UserAuth`、`OrderHub`、`AssetLedger`、`RecFilter` 四个模块，并通过接口依赖与事件发布解耦。

## 审查表

| SOLID 原则 | 检查问题 | AI 设计是否违反 | 违反说明 | 修正方案 |
|-----------|---------|--------------|---------|---------|
| S - 单一职责 | 有没有类承担了过多职责？ | 是 | `CampusHubSystem` 同时负责认证、发布、接单、结算、风控、推荐、通知、仲裁，类的变化原因过多。 | 拆分为 `OrderApplicationService`、`ContentModerationService`、`NotificationDispatcher`、`WalletAccount`、`ReputationProfile` 等，并按模块归位。 |
| O - 开闭原则 | 新增需求类型是否需要修改现有代码？ | 是 | 若新增“自动结算规则”或“新的推荐规则”，只能直接修改 `CampusHubSystem` 中原有流程。 | 引入 `SettlementPolicy`、`RecommendationStrategy` 等接口，通过增加实现类扩展行为，而不是改动主流程。 |
| L - 里氏替换 | 子类是否可以替换父类使用？ | 是 | 原始设计没有明确抽象，若后续把不同订单状态做成继承层次，极易出现某些状态不支持某些操作而抛异常，破坏替换性。 | 使用 `OrderState` 接口约束合法行为，让各状态类仅实现允许的状态迁移，主流程依赖抽象状态对象。 |
| I - 接口隔离 | 有没有接口太“胖”，包含不需要的方法？ | 是 | 原始设计没有边界接口，唯一服务对象 `CampusHubSystem` 暴露了一整套庞大方法，调用方被迫依赖自己根本用不到的能力。 | 按场景拆出 `UserIdentityGateway`、`ModerationGateway`、`AssetTransferService`、`DomainEventPublisher` 等窄接口。 |
| D - 依赖倒转 | 高层模块是否直接依赖了低层模块的具体实现？ | 是 | 业务入口直接操作用户、订单、积分、通知等具体数据结构，没有稳定抽象层，高层流程与底层实现强耦合。 | 让 `OrderApplicationService` 依赖接口而非具体类；推荐、风控、结算等能力由实现模块注入。 |

## 逐项说明

### 1. 单一职责问题

AI 初版把“一个平台应该做的所有事”直接映射成一个类，这是生成式设计里最常见的问题。这样的类在真实开发中会导致：

- 任何需求改动都可能影响同一个类
- 单元测试难以隔离
- 团队多人协作时容易产生代码冲突

修正后我们以“变化原因”而不是“页面功能”拆类，让身份、订单、账本、风控分别演化。

### 2. 开闭原则问题

校园互助平台后续很可能新增：

- 新的推荐策略
- 新的结算策略
- 新的通知渠道
- 新的风控规则

如果不提前抽象策略接口，每次新增都得改核心流程类，风险会越来越高。修正版中将波动点提炼成策略/网关接口，避免主流程膨胀。

### 3. 里氏替换问题

订单流转天然带有状态约束。例如：

- `PendingAcceptanceState` 允许接单，不允许确认完成
- `PendingConfirmationState` 允许确认或自动结算，不允许再次接单

如果只是用一个大类加一堆 `if-else`，后续抽继承层次时容易出现“不该支持的方法也暴露出来”的问题。修正版直接把状态行为抽象出来，降低替换风险。

### 4. 接口隔离问题

P2 文档已经明确指出模块间不能直接传递所有内部对象，更不能跨模块直接访问数据库。原始设计却让外部调用者面向一个全能服务对象，这与架构边界冲突。修正版中的接口都是按实际调用场景切分的，调用方只依赖自己真正需要的方法。

### 5. 依赖倒转问题

AI 初版是典型的“业务类直接抓具体实现”。这种设计的直接后果是：

- 无法替换实现
- 测试桩难写
- 模块边界名存实亡

修正版让 `OrderHub` 依赖 `UserIdentityGateway`、`ModerationGateway`、`AssetTransferService` 等抽象契约，符合 P2 的“进程内接口通信”要求。
