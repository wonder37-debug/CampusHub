# API 规范（OpenAPI）审查报告 — AI 辅助实验

下面是API规范文档审查中，发现的一些问题。

1) 命名一致性
- 问题：部分 schema 名称混杂下划线（例如 `StandardResponse_Auth`）。
- 建议：统一风格，OpenAPI 中 schema 使用 PascalCase （`DemandListResponse`、`AuthResponse` 等），路径保持小写复数。

2) 错误处理与状态码
- 问题：当前多数成功响应使用 HTTP 200，错误示例给出自定义 `code` 字段，但没有充分使用标准 HTTP 状态码（400/401/403/404/409/500）。
- 建议：在 OpenAPI 中明确列出 4xx/5xx 状态码响应体（400 -> 参数校验，401 -> 认证失败，403 -> 权限不足，404 -> 资源未找到，409 -> 冲突）。同时保留 `code` 字段用于业务级错误码。

3) 参数校验不完整
- 问题：在 YAML 中 `DemandCreate` 指定了必填字段，但未对字符串长度、reward 的最小值、时间范围约束等做校验。
- 建议：补充 `minLength`/`maxLength`、`minimum`/`maximum`、`format: date-time`、并在文档中说明字段约束与错误码映射。

4) 安全与速率限制
- 问题：未在规范中声明速率限制、CORS 策略、与数据脱敏/审计要求。
- 建议：在实现说明里加入速率限制建议（如每用户每分钟 60 次），对敏感字段写明存储与传输要求（HTTPS 强制、日志脱敏）。

5) 分页与排序的一致性
- 问题：列表返回使用 `page/size`，但未说明最大 `size` 与默认排序行为。
- 建议：明确 `size` 最大值（例如 100），并在 API 规范中定义 `sort` 的可选枚举和默认排序（例如默认按 `startTime desc`）。

6) 并发与幂等性
- 问题：接单操作可能并发导致重复接单；OpenAPI 中未指定幂等性或并发控制方式。
- 建议：标注非幂等操作（POST /demands/{id}/accept），并在实现指南中要求使用数据库事务+乐观锁/行锁或利用 CAS 逻辑，返回 409 表示冲突。

7) 可扩展性与分页游标
- 建议：对于长期扩展场景，考虑提供基于游标的分页（cursor）以改善效率，同时保留 page/size 简单模式。

基于上述问题，我们让AI重新修补了API规范文档，接口后续可能还存在问题，留待开发时注意。
