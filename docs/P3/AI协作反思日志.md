## 2. AI 助力分析（Q1）

本阶段 AI 的主要作用是快速生成结构化初稿，并帮助团队把不同交付物统一到同一套领域命名中。尤其在类图、API 和数据库设计三个部分，AI 可以先输出一版完整骨架，再由团队人工修正边界、约束和一致性问题，整体上显著减少了从零开始组织文档的时间。

具体来看，AI 在以下方面帮助最大：

1. 快速把业务需求映射到类、接口和表结构。
2. 帮助统一命名，例如用户、需求、订单、评价、通知、资产流水等核心概念。
3. 为 API 文档和 ER 图提供第一版可读结构，便于团队直接审查与修改。
4. 通过文档化输出暴露设计缺口，例如分页、权限、并发冲突和索引约束。

## 3. AI 误导分析（Q2）

SOLID 检查中发现的 AI 设计问题数量：5

最严重的设计问题是：把发布、接单、结算、推荐、风控、通知、仲裁全部堆进一个大类，导致职责边界不清、修改面过大，并且和 P2 的模块化边界冲突。

其他典型问题包括：

1. 没有把状态流转抽象成独立状态模型，容易用大量 `if-else` 维护订单生命周期。
2. 没有把通知、结算和审计作为独立副作用处理，主业务流程过于臃肿。
3. 接口设计偏大，调用方容易依赖不需要的方法，不符合接口隔离原则。
4. 直接依赖具体实现而不是仓储或策略抽象，导致依赖倒转失败。

## 4. Prompt 改进计划（Q3）

上阶段改进措施的验证结果：

将需求拆成“模块边界 + 输入输出 + 约束条件”之后，AI 生成内容的结构明显更稳定，类图和 API 文档的耦合问题减少了，后续修正也更集中在少数关键约束上。简单来说，少量的、精细的、多轮的prompt对AI来说更友好（当然对人也一样），更容易获得我们想要的结果。

本阶段新的改进措施：

1. 让 AI 先输出职责表，再输出类图和接口清单，减少一开始就把系统画大的倾向。
2. 在 Prompt 中明确写出并发、权限、分页、错误码和索引要求，避免文档缺少关键工程约束。
3. 对数据库设计增加“第三范式 + 索引 + 审计”的固定检查项，先审结构再审语法。

## 5. 本阶段的核心工程判断

决策内容：接单流程必须采用事务控制与并发冲突处理，不能只依赖“先查再写”的表面逻辑。

为什么 AI 无法做这个决策：

AI 可以生成流程模板，但无法代替团队对并发一致性、数据库锁粒度、业务冲突边界和失败后果的工程判断。接单、评价和资金流水都属于高一致性场景，必须结合系统实际负载和故障容忍度由人来决定实现策略。
