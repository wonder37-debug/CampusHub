# CampusHub 后端待修复清单

本文基于 [CampusHub 问题报告](./CampusHub%20问题报告.md) 与根目录 `change_log.md` 的交叉核对结果整理，目标是只保留“当前后端服务层仍未完成”或“虽然有实现，但仍未满足问题报告要求”的事项，便于后续直接排期与开发。

## 说明

- 范围：仅关注后端服务层、领域规则、通知链路、权限校验与对应测试缺口。
- 判定标准：
  - `已完成`：`change_log.md` 中有明确记录，且代码实现已覆盖问题报告要求。
  - `部分完成`：已有实现，但与问题报告要求仍有差距。
  - `未完成`：服务层尚无对应规则或链路。
- 本清单默认不讨论纯前端文案、样式、跳转交互问题。

## P0 待修复

### 1. 过期需求自动下架与禁止接单

- 对应问题：`P0-01`
- 当前状态：`未完成`
- 影响：
  - 过期需求仍可能出现在公开列表中。
  - 只要状态还是 `PENDING`，用户仍可成功接单。
  - 平台缺少“过期”这个明确的业务状态。
- 当前代码现状：
  - [backend/src/main/java/com/campushub/backend/demand/domain/DemandStatus.java](/E:/homeworki/sec-ii-2026/backend/src/main/java/com/campushub/backend/demand/domain/DemandStatus.java:3) 没有 `EXPIRED` 状态。
  - [backend/src/main/java/com/campushub/backend/demand/service/DemandApplicationServiceImpl.java](/E:/homeworki/sec-ii-2026/backend/src/main/java/com/campushub/backend/demand/service/DemandApplicationServiceImpl.java:81) 的公开列表直接基于 `findAll()` 过滤，没有排除过期需求。
  - [backend/src/main/java/com/campushub/backend/order/service/OrderApplicationServiceImpl.java](/E:/homeworki/sec-ii-2026/backend/src/main/java/com/campushub/backend/order/service/OrderApplicationServiceImpl.java:52) 的接单逻辑没有校验 `endTime < now`。
- 建议修复：
  - 在 `DemandStatus` 中新增 `EXPIRED`。
  - 在需求服务层增加“按时间推进状态”的逻辑。
  - 增加定时任务，周期性将 `endTime < now && status == PENDING` 的需求更新为 `EXPIRED`。
  - 在接单前追加兜底校验：若需求已过期，直接返回业务冲突。
  - 在公开列表默认排除 `EXPIRED`。
- 建议补测：
  - 过期需求不会出现在公开列表。
  - 过期需求接单返回冲突错误。
  - 定时任务执行后，`PENDING -> EXPIRED` 状态正确落库。

### 2. 信用分计算公式仍会单次暴跌

- 对应问题：`P0-02`
- 当前状态：`未完成`
- 影响：
  - 单次低分评价会造成信用分快速、大幅下降。
  - 易受恶意差评影响，不符合“平滑更新”的预期。
- 当前代码现状：
  - [backend/src/main/java/com/campushub/backend/review/service/ReviewApplicationServiceImpl.java](/E:/homeworki/sec-ii-2026/backend/src/main/java/com/campushub/backend/review/service/ReviewApplicationServiceImpl.java:106) 仍采用 `平均评分 * 20` 直接映射到百分制。
  - 该公式下，一条 1 星评价会让信用分趋近 `20`，与问题报告要求不符。
- 建议修复：
  - 改为加权平滑公式，或引入“历史均值 + 单次变动上限”的方案。
  - 明确单次评价对信用分的最大影响幅度。
  - 将评分规则沉淀为可读常量或独立方法，便于后续调整。
- 建议补测：
  - 从 `100` 分开始收到一次 `1` 星评价，不应暴跌到 `20`。
  - 连续多次评分后，信用分变化应符合新公式预期。
  - 无历史评价用户的默认分值逻辑保持正确。

### 3. 发布需求缺少余额充足校验

- 对应问题：`P0-06`
- 当前状态：`未完成`
- 影响：
  - 用户可以发布超过当前余额的有偿需求。
  - 后续结算、冻结或扣款流程会失真，形成资金逻辑漏洞。
- 当前代码现状：
  - [backend/src/main/java/com/campushub/backend/demand/service/DemandApplicationServiceImpl.java](/E:/homeworki/sec-ii-2026/backend/src/main/java/com/campushub/backend/demand/service/DemandApplicationServiceImpl.java:48) 的发布流程未校验发布者余额。
  - [backend/src/main/java/com/campushub/backend/demand/service/DemandApplicationServiceImpl.java](/E:/homeworki/sec-ii-2026/backend/src/main/java/com/campushub/backend/demand/service/DemandApplicationServiceImpl.java:259) 仅校验 `reward >= 0`。
  - 用户领域对象已经有 `balance` 与 `frozenBalance` 字段，但服务层尚未使用。
- 建议修复：
  - 发布需求时校验 `reward <= balance - frozenBalance` 或平台定义的“可用余额”。
  - 若后续要做担保/冻结，建议在发布成功时同步处理冻结余额。
  - 给出明确业务错误码和提示语义，如“余额不足，无法发布该需求”。
- 建议补测：
  - 余额不足时发布失败。
  - 余额刚好等于报酬时可正常发布。
  - 免费需求或 `reward = 0` 不受影响。

## P1 待修复

### 4. 公开需求列表没有限制只展示可公开需求

- 对应问题：`P1-05`
- 当前状态：`未完成`
- 影响：
  - `REVIEWING` 等非公开需求仍可能进入首页大厅。
  - 审核中的内容对普通用户暴露，违背业务规则。
- 当前代码现状：
  - [backend/src/main/java/com/campushub/backend/demand/service/DemandApplicationServiceImpl.java](/E:/homeworki/sec-ii-2026/backend/src/main/java/com/campushub/backend/demand/service/DemandApplicationServiceImpl.java:86) 公开列表没有按需求状态做默认过滤。
  - [backend/src/main/java/com/campushub/backend/demand/dto/DemandQuery.java](/E:/homeworki/sec-ii-2026/backend/src/main/java/com/campushub/backend/demand/dto/DemandQuery.java:7) 也没有状态维度，意味着当前公开查询无法表达“只看 `PENDING`”。
- 建议修复：
  - 将公开列表默认限定为 `status == PENDING`。
  - 如果后续要支持发布者本人或管理员查看非公开需求，建议拆分“公开列表接口”和“管理/我的需求接口”。
  - 若保留统一查询对象，可增加显式状态参数，但公开接口仍应做服务层兜底。
- 建议补测：
  - `REVIEWING` 需求不会出现在公开列表。
  - `PENDING` 需求可正常展示。
  - 已取消、已完成、已过期需求不会进入公开大厅。

### 5. 管理员发布/接单权限边界未收紧

- 对应问题：`P1-04`
- 当前状态：`未完成`
- 影响：
  - 管理员账号仍可参与普通用户业务流。
  - 角色边界模糊，容易引发越权和测试歧义。
- 当前代码现状：
  - [backend/src/main/java/com/campushub/backend/demand/service/DemandApplicationServiceImpl.java](/E:/homeworki/sec-ii-2026/backend/src/main/java/com/campushub/backend/demand/service/DemandApplicationServiceImpl.java:166) 只校验封禁状态，不限制 `ADMIN` 发布需求。
  - [backend/src/main/java/com/campushub/backend/order/service/OrderApplicationServiceImpl.java](/E:/homeworki/sec-ii-2026/backend/src/main/java/com/campushub/backend/order/service/OrderApplicationServiceImpl.java:52) 接单逻辑也未限制管理员身份。
- 建议修复：
  - 在发布需求和接单服务入口统一校验：管理员不可参与普通交易流程。
  - 如后续存在“超管代操作”场景，应单独设计后台专用接口，而不是复用用户侧接口。
- 建议补测：
  - 管理员发布需求失败。
  - 管理员接单失败。
  - 普通用户既有流程不受影响。

### 6. 发布需求后未通知管理员审核

- 对应问题：`P1-09`
- 当前状态：`未完成`
- 影响：
  - 新需求创建后，管理员无法及时收到待审核提醒。
  - 审核流程依赖人工刷新后台，不符合问题报告要求。
- 当前代码现状：
  - [backend/src/main/java/com/campushub/backend/demand/service/DemandApplicationServiceImpl.java](/E:/homeworki/sec-ii-2026/backend/src/main/java/com/campushub/backend/demand/service/DemandApplicationServiceImpl.java:48) 发布成功后没有任何管理员通知逻辑。
  - [backend/src/main/java/com/campushub/backend/notification/domain/NotificationType.java](/E:/homeworki/sec-ii-2026/backend/src/main/java/com/campushub/backend/notification/domain/NotificationType.java:3) 不包含 `REVIEW_REQUEST`。
  - [backend/src/main/java/com/campushub/backend/notification/service/NotificationApplicationService.java](/E:/homeworki/sec-ii-2026/backend/src/main/java/com/campushub/backend/notification/service/NotificationApplicationService.java:5) 也没有对应通知方法。
- 建议修复：
  - 增加 `NotificationType.REVIEW_REQUEST`。
  - 在需求发布成功后，查询 `role = ADMIN` 的用户并批量发送站内通知。
  - `relatedId` 指向需求 ID，方便前端跳转审核页。
- 建议补测：
  - 用户发布需求后，所有管理员收到一条审核通知。
  - 通知类型、`relatedId`、正文内容正确。
  - 无管理员时不应抛异常。

### 7. 个人资料昵称唯一性后端校验缺失

- 对应问题：`P1-08`
- 当前状态：`未完成`
- 影响：
  - 平台可出现重复昵称，影响识别与沟通。
  - 前端即使做了提示，也无法替代后端约束。
- 当前代码现状：
  - [backend/src/main/java/com/campushub/backend/auth/service/AuthApplicationServiceImpl.java](/E:/homeworki/sec-ii-2026/backend/src/main/java/com/campushub/backend/auth/service/AuthApplicationServiceImpl.java:130) 更新资料时直接保存昵称，没有查重。
  - [backend/src/main/java/com/campushub/backend/auth/repository/UserRepository.java](/E:/homeworki/sec-ii-2026/backend/src/main/java/com/campushub/backend/auth/repository/UserRepository.java:8) 当前也没有昵称查找接口。
- 建议修复：
  - 在仓储层增加按昵称查询能力。
  - 更新资料时忽略本人后做唯一性校验。
  - 统一注册与资料更新时的昵称规则，避免一个入口有限制，另一个入口没有。
- 建议补测：
  - 修改成别人已使用的昵称时失败。
  - 修改为自己的原昵称时允许。
  - 空昵称回退默认昵称时的逻辑仍然正确。

## 部分完成但仍建议继续修正

### 8. 管理后台统计接口已实现，但 DAU 口径仍偏弱

- 对应问题：`P0-03`
- 当前状态：`部分完成`
- 说明：
  - `change_log.md` 已记录管理后台统计能力上线。
  - [backend/src/main/java/com/campushub/backend/admin/service/AdminApplicationServiceImpl.java](/E:/homeworki/sec-ii-2026/backend/src/main/java/com/campushub/backend/admin/service/AdminApplicationServiceImpl.java:154) 也确实实现了统计接口。
- 当前差距：
  - [backend/src/main/java/com/campushub/backend/admin/service/AdminApplicationServiceImpl.java](/E:/homeworki/sec-ii-2026/backend/src/main/java/com/campushub/backend/admin/service/AdminApplicationServiceImpl.java:232) 的 DAU 判定依赖 `createdAt`/`updatedAt` 是否为今天。
  - 这和问题报告中建议的“今日登录记录或操作记录”口径并不一致，容易统计失真。
- 建议修复：
  - 如果后端已有登录日志/行为日志，应改用真实活跃行为计算 DAU。
  - 如果暂时没有登录日志，建议至少基于订单、需求、通知已读、评论等真实业务操作构建活跃口径。
- 建议补测：
  - 仅修改资料的用户是否算活跃，需要与产品定义一致。
  - 今天无登录无操作、但历史创建过账号的用户，不应被统计为 DAU。
