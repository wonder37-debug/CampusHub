# P4 协作与接口规范

> 目标：让数据访问层、业务逻辑层、展示层并行开发时，目录清晰、接口稳定、联调成本可控。

## 1. 是否需要建文件夹

需要。不是为了“分得很细”，而是为了让职责边界清晰，避免多人同时改同一个目录。

### 后端建议目录

```text
backend/src/main/java/com/campushub/backend/
  config/
  controller/
  dto/
  entity/
  exception/
  mapper/
  repository/
  service/
  service/impl/
  security/
  util/
```

### 前端建议目录

```text
frontend/src/
  api/
  assets/
  components/
  composables/
  router/
  stores/
  types/
  utils/
  views/
```

### 目录使用原则

- `controller` 只做请求接入、参数校验、返回包装，不写复杂业务。
- `service` 承担业务规则、状态流转、权限判断。
- `repository` 或 `mapper` 只负责数据库读写。
- `dto` 只放请求/响应对象，不直接暴露数据库实体。
- 前端 `api` 统一封装请求，`views` 负责页面，`components` 负责可复用组件。
- 不要把数据库实体直接传到前端页面层。

## 2. 层间对接规范

### 后端分层调用关系

```text
controller -> service -> repository/mapper -> database
```

### 约定

- `controller` 不直接调用数据库层。
- `service` 不关心 HTTP 细节，只处理业务结果。
- `repository/mapper` 不写业务判断。
- 每个层的入参、出参都尽量用明确的 DTO 或 VO。

### 推荐的请求返回格式

```json
{
  "code": 0,
  "message": "OK",
  "data": {}
}
```

错误返回示例：

```json
{
  "code": 40001,
  "message": "参数校验失败",
  "errors": {
    "title": "标题不能为空"
  }
}
```

### 错误处理约定

- 参数错误统一返回 400。
- 未登录返回 401。
- 无权限返回 403。
- 资源不存在返回 404。
- 业务冲突返回 409，例如重复接单。

## 3. 必须先统一的接口

你们四个组员至少先对齐下面这些接口，再开始并行开发。

### 用户管理
- `POST /api/v1/auth/register`
- `POST /api/v1/auth/login`
- `GET /api/v1/users/{userId}`
- `PUT /api/v1/users/{userId}`

### 需求模块
- `POST /api/v1/demands`
- `GET /api/v1/demands`
- `GET /api/v1/demands/{demandId}`
- `PUT /api/v1/demands/{demandId}`
- `DELETE /api/v1/demands/{demandId}`
- `POST /api/v1/demands/{demandId}/accept`
- `POST /api/v1/demands/{demandId}/cancel`

### 订单模块
- `GET /api/v1/orders/{orderId}`
- `POST /api/v1/orders/{orderId}/confirm`
- `POST /api/v1/orders/{orderId}/start`
- `POST /api/v1/orders/{orderId}/complete`
- `POST /api/v1/orders/{orderId}/cancel`
- `GET /api/v1/users/{userId}/orders`

### 评价与通知
- `POST /api/v1/orders/{orderId}/reviews`
- `GET /api/v1/notifications`

## 5. 状态流转约定

### 需求状态
- `draft`：草稿
- `open`：可接单
- `accepted`：已接单
- `cancelled`：已取消
- `closed`：已完成

### 订单状态
- `accepted`
- `confirmed`
- `in_progress`
- `completed`
- `cancelled`

状态流转必须写清楚谁能改、何时能改、失败返回什么。

## 6. 联调前置要求

在正式联调前，每个组员都要保证：

- 接口路径已经确认。
- 请求参数和响应字段已经固定。
- 失败场景已经约定。
- 认证方式已经统一为 JWT Bearer Token。
- 日期格式统一为 ISO8601。
- 枚举值统一，不要前后端各写一套。

## 7. 推荐的协作方式

### 7.1 层与目录的对应关系

下面这张表就是你们最该按它分工的版本，谁负责哪一层，就主要写对应目录里的文件。

| 负责层 | 主要负责的人 | 主要写哪些目录 | 主要产出 |
|------|------|------|------|
| 数据访问层 | 数据访问层负责人 | `entity/`、`mapper/`、`repository/`、`sql/`（如有） | 实体类、Mapper/Repository、SQL 初始化脚本、查询条件 |
| 业务逻辑层 | 业务逻辑层负责人 | `service/`、`service/impl/`、`dto/`、`exception/`、`security/`（部分） | Service 接口、Service 实现、业务 DTO、异常定义、状态流转逻辑 |
| 展示层 | 展示层负责人 | `frontend/src/api/`、`frontend/src/views/`、`frontend/src/components/`、`frontend/src/types/`、`frontend/src/stores/` | 页面组件、请求封装、前端类型、状态管理、交互逻辑 |

### 7.2 细分到文件级别的分工建议

- 数据访问层负责人主要写：`backend/src/main/java/com/campushub/backend/entity/`、`backend/src/main/java/com/campushub/backend/mapper/`、`backend/src/main/java/com/campushub/backend/repository/`，以及建表 SQL。
- 业务逻辑层负责人主要写：`backend/src/main/java/com/campushub/backend/service/`、`backend/src/main/java/com/campushub/backend/service/impl/`、`backend/src/main/java/com/campushub/backend/dto/`、`backend/src/main/java/com/campushub/backend/exception/`。
- 如果后端需要统一返回、鉴权或拦截器，通常也由业务逻辑层负责人维护 `backend/src/main/java/com/campushub/backend/security/`、`backend/src/main/java/com/campushub/backend/config/` 里的相关配置。
- 展示层负责人主要写：`frontend/src/api/` 里的接口封装，`frontend/src/views/` 里的页面，`frontend/src/components/` 里的复用组件，`frontend/src/types/` 里的类型定义。
- 如果前端有全局状态管理，再由展示层负责人维护 `frontend/src/stores/`；如果有通用工具函数，再维护 `frontend/src/utils/`。

### 7.3 你们四个人最清晰的分配方式

如果四个人已经固定分工，建议这样落地：

- 人员 A：数据访问层负责人，负责 `entity/`、`mapper/`、`repository/`、SQL。
- 人员 B：业务逻辑层负责人，负责 `service/`、`service/impl/`、`dto/`、`exception/`。
- 人员 C：展示层负责人，负责 `api/`、`views/`、`components/`、`types/`。
- 人员 D：联调/集成负责人，负责 `config/`、`security/`、接口契约文档、联调记录、测试数据和 Bug 修复日志。

这样分的好处是：每个人写的目录尽量不重叠，联调时只需要对接口和 DTO，不需要互相改核心目录。

### 数据访问层负责人
负责：
- 表结构
- CRUD
- 索引
- 查询条件

交付物：
- 实体类
- Mapper/Repository
- SQL 初始化脚本

### 业务逻辑层负责人
负责：
- 注册登录规则
- 接单与订单流转
- 评价和通知触发

交付物：
- Service 接口
- Service 实现
- 单元测试

### 展示层负责人
负责：
- 页面布局
- 表单交互
- 列表渲染
- 接口调用封装

交付物：
- 页面组件
- API 调用封装
- 类型定义

## 8. 接口变更流程

1. 先改接口规范文档。
2. 再同步后端 DTO 和前端类型。
3. 最后改实现代码。
4. 变更后必须补充联调说明。

不要先改代码、后补文档，否则容易出现“代码能跑但对不上”的情况。

## 9. 适合你们组的最低建议

如果时间紧，至少建立这三类目录：

- 后端：`controller`、`service`、`repository`
- 前端：`api`、`views`、`components`
- 文档：`docs/P4/接口对接规范.md`

## 10. 可直接复用的接口对接模板

建议你们把每个接口都按下面格式写进同一份文档里，前后端都只认这份约定。

### 接口定义模板

| 项目 | 内容 |
|------|------|
| 接口名称 | 例如：用户登录 |
| URL | `POST /api/v1/auth/login` |
| 负责人 | 后端：___，前端：___，数据：___ |
| 鉴权 | 是否需要 JWT，是否需要管理员权限 |
| 请求参数 | 字段名、类型、是否必填、说明、校验规则 |
| 响应成功 | `code`、`message`、`data` 的结构 |
| 响应失败 | `400/401/403/404/409/500` 分别怎么返回 |
| 数据库影响 | 是否新增/更新/查询哪些表 |
| 状态流转 | 是否会改变需求/订单/通知状态 |
| 联调说明 | 前端传什么、后端回什么、用什么测试数据 |

### 单接口示例

#### 用户登录

| 项目 | 内容 |
|------|------|
| 接口名称 | 用户登录 |
| URL | `POST /api/v1/auth/login` |
| 负责人 | 后端：业务逻辑层，前端：展示层，数据：无 |
| 鉴权 | 否 |
| 请求参数 | `studentId` string 必填；`password` string 必填，最少 8 位 |
| 响应成功 | `code=0`，`data.token`、`data.expiresIn`、`data.user` |
| 响应失败 | `400` 参数错误；`401` 认证失败；`500` 系统异常 |
| 数据库影响 | 查询 `users` 表，校验密码哈希 |
| 状态流转 | 无 |
| 联调说明 | 前端成功后保存 token，并放入后续请求头 |

### 建议的接口联调流程

1. 后端先把接口文档定死，前端按文档写请求封装。
2. 数据访问层先提供实体、表结构和 SQL 初始化脚本。
3. 业务层先完成核心状态流转，再补异常分支。
4. 前端先接 mock 数据，确认字段后再连真实接口。
5. 每次接口变更都要同步更新这份文档和 OpenAPI。

## 11. 如果你们想更稳妥的落地方式

如果你们四个人已经明确分工，建议再加一个很小的“对接契约目录”：

```text
docs/P4/contracts/
  auth.md
  demand.md
  order.md
  review.md
  notification.md
```

每个文件只写一个模块的接口表、状态流转、错误码和示例 JSON。这样联调时查找最快，也最不容易乱。

这样就足够支持四个人并行开发和后期联调。
